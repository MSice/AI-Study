## 7.1 模型微调（Fine-tuning）

### 7.1.1 OpenAI Fine-tuning 文档

> 原文：[OpenAI Fine-tuning 文档](https://platform.openai.com/docs/guides/fine-tuning)（英文）

OpenAI 的微调文档是入门模型微调最直接的资源。它把微调放在"模型优化工作流"的大框架下，和评估（Evals）、提示词工程并列，形成一套完整的优化闭环。

#### 微调是什么？为什么需要它？

微调就像"给通才做专项培训"。OpenAI 的基座模型已经能处理各种任务，但微调可以让你用**自己的数据**对模型做额外训练，让它在**特定任务**上表现更好。

文档明确指出，微调相比单纯用提示词，能带来这些好处：

| 好处 | 说明 |
|------|------|
| **用小模型做大事** | 可以把更小、更便宜、更快的模型训练成在特定任务上媲美大模型，降低成本 |
| **数据不出门** | 可以用私有或敏感数据训练，而不必每次请求都把数据塞进提示词里 |
| **省 Token、省延迟** | 微调后可以用更短的提示词、更少的示例，节省 Token 费用并降低延迟 |
| **突破上下文限制** | 可以喂给模型比单次请求能容纳的更多的输入-输出示例，让模型处理更多样化的输入 |

#### 微调工作流：评估 → 提示词 → 微调 → 再评估

文档强调，模型优化是一个**持续迭代的飞轮**：

1. **先写评估**：用测试数据定义"什么是好结果"，建立基线
2. **用提示词试**：给模型上下文和指令，看效果如何
3. **必要时微调**：如果提示词不够，再考虑微调
4. **根据评估反馈调整**：改提示词或改训练数据
5. **重复**：不断循环，持续改进

#### 微调方法一览

OpenAI 目前支持多种微调方法，适合不同场景：

| 方法 | 原理 | 适合场景 | 支持模型 |
|------|------|---------|---------|
| **监督微调（SFT）** | 提供"输入-正确输出"的示例，让模型学会按期望方式回答 | 纠正指令遵循失败、特定格式输出、 nuanced 翻译、分类 | gpt-4.1 系列、gpt-4.1-mini、gpt-4.1-nano |
| **直接偏好优化（DPO）** | 提供正确和错误两个回答，标注哪个更好，让模型学会偏好 | 摘要聚焦重点、聊天消息的语气和风格 | 同上 |
| **强化微调（RFT）** | 生成回答 → 专家打分 → 强化高分回答的推理链 | 法律案例检索、医学诊断、复杂领域推理 | 仅推理模型（如 o4-mini） |
| **视觉微调** | 提供图片输入做监督微调 | 图片分类、复杂提示下的指令遵循 | gpt-4o |

#### 微调的基本流程

1. **收集数据**：准备符合格式的输入-输出示例
2. **格式化为 JSONL**：按 OpenAI 要求的格式整理
3. **上传到 OpenAI**：通过 API 或控制台上传
4. **创建微调任务**：选择方法，开始训练
5. **（RFT 需额外）定义 Grader**：用专家评分器给模型输出打分
6. **评估结果**：在测试集上验证效果

文档建议从**监督微调（SFT）**开始，这是最直观、最常用的方式。

### 7.1.2 Unsloth：高效本地微调工具

> 原文：[Unsloth](https://github.com/unslothai/unsloth)（GitHub）

Unsloth 是一个面向**本地微调**的开源工具，主打"更快、更省显存"。如果你不想用 OpenAI 的云端微调，而是想在自己的 GPU 上微调开源模型（如 Llama、Qwen、Gemma、DeepSeek 等），Unsloth 是一个很好的选择。

#### 核心卖点

- **训练速度约 2 倍**：通过优化计算和内存访问，显著缩短微调时间
- **显存占用约减少 70%**：在同样显存下可以微调更大的模型，或用更小的显卡完成微调
- **支持多种开源模型**：Llama、Qwen、Gemma、DeepSeek、Mistral 等
- **支持 LoRA**：采用低秩适配（LoRA）等高效微调方法，只训练少量参数

#### 适合谁用？

- 有本地 GPU、想微调开源模型的开发者
- 数据敏感、不想把数据上传到云端的场景
- 需要快速迭代、多次实验微调效果的团队

#### 与 OpenAI 微调的对比

| 维度 | OpenAI Fine-tuning | Unsloth |
|------|-------------------|---------|
| 模型 | GPT 系列（闭源） | 开源模型（Llama、Qwen 等） |
| 运行位置 | 云端 | 本地 GPU |
| 数据隐私 | 数据需上传 | 数据不出本机 |
| 成本 | 按训练和推理计费 | 主要是一次性硬件成本 |
| 上手难度 | 最简单（API 调用） | 需要配置环境和依赖 |

## 7.2 模型评估与基准测试

### 7.2.1 Ragas：RAG 评估框架

> 原文：[Ragas 文档](https://docs.ragas.io/)（英文）

Ragas 是一个专门用于**评估 RAG（检索增强生成）系统**的 Python 库。它的核心思想是：传统评估指标无法衡量 LLM 应用真正重要的东西，而人工评估又无法规模化。Ragas 用"LLM 驱动的指标 + 系统化实验"来建立持续改进的评估闭环。

#### 为什么需要 Ragas？

- **告别"感觉还行"**：从" vibe check"（凭感觉判断）转向可量化的评估
- **自动化评估**：不需要大量人工标注，就能系统评估 RAG 效果
- **实验驱动**：支持做实验、对比不同配置、追踪结果，形成迭代闭环

#### 核心指标（按 RAG 组件划分）

Ragas 把指标分为**检索器指标**和**生成器指标**：

**检索器指标（衡量"找得准不准"）**：

| 指标 | 含义 | 怎么算 |
|------|------|--------|
| **Context Precision（上下文精度）** | 检索到的文档中，相关的是否排在前面 | 相关文档的排名越靠前，分数越高 |
| **Context Recall（上下文召回）** | 检索到的文档是否覆盖了回答问题所需的信息 | 标准答案中的关键信息有多少被检索到 |

**生成器指标（衡量"答得好不好"）**：

| 指标 | 含义 | 怎么算 |
|------|------|--------|
| **Faithfulness（忠实度）** | 回答是否基于检索到的上下文，有没有瞎编 | 把回答拆成多个"主张"，看每个主张能否从上下文中推出；分数 = 有依据的主张数 / 总主张数 |
| **Answer Relevancy（回答相关性）** | 回答是否切题 | 用"能产生同样回答的问题"与真实问题的相似度来衡量 |

#### 使用方式

- **实验优先**：用 `experiments` 做配置对比，观察指标变化
- **自定义指标**：用装饰器定义自己的评估指标，或使用内置指标库
- **易集成**：支持 LangChain、LlamaIndex 等框架，内置数据集管理和结果追踪

#### 典型工作流

1. 准备测试数据集（问题 + 标准答案 + 检索到的上下文 + 模型回答）
2. 选择或定义要评估的指标
3. 运行评估，得到各指标的分数
4. 调整 RAG 配置（如检索策略、提示词），再跑评估
5. 对比不同配置的效果，持续迭代

### 7.2.2 LMSYS Chatbot Arena：基于人类偏好的模型评估

> 原文：[LMSYS Chatbot Arena](https://chat.lmsys.org/)（英文）

LMSYS Chatbot Arena 是由加州大学伯克利分校团队维护的**人类盲测投票平台**，是目前最权威的 LLM 综合能力排行榜之一。

#### 评测方式

1. 用户提出一个问题
2. 系统随机选两个**匿名**模型来回答（用户不知道是哪两个）
3. 用户看完两个回答后，投票选出更好的那个
4. 用 **Elo 评分**（类似国际象棋排名）根据大量投票结果给模型排名

这种"盲测 + 人类投票"的方式，比纯自动化评测更接近真实使用体验。

#### 怎么用？

- **看排行榜**：了解各模型的综合实力和在不同类别（编程、数学、创意写作等）的表现
- **亲自投票**：参与评测，为排行榜贡献数据
- **选模型参考**：在做技术选型时，可结合 Arena 排名和业务需求做决策

#### 与 Ragas 的对比

| 维度 | Ragas | LMSYS Chatbot Arena |
|------|-------|---------------------|
| 评估对象 | RAG 系统、特定应用 | 通用对话模型 |
| 评估方式 | 自动化指标 | 人类盲测投票 |
| 粒度 | 可针对检索、生成等组件 | 整体对话质量 |
| 适用场景 | 优化自己的 RAG 应用 | 了解模型大盘、选型参考 |

## 7.3 模型部署与推理优化

### 7.3.1 vLLM：高性能推理引擎

> 原文：[vLLM 官方文档](https://docs.vllm.ai/)（英文）

vLLM 是当前最流行的高性能 LLM 推理引擎之一，由加州大学伯克利分校 Sky Computing Lab 发起，现已发展为社区驱动的开源项目。它的口号是：**Easy, fast, and cheap LLM serving for everyone**（简单、快速、便宜的 LLM 服务，人人可用）。

#### 核心优势：快

vLLM 通过多项技术实现**高吞吐、低延迟**：

| 技术 | 作用 |
|------|------|
| **PagedAttention** | 借鉴操作系统虚拟内存的思路，把 KV Cache（注意力计算的中间结果）分成小块"页"来管理，大幅减少显存浪费。传统方案可能浪费 60–80% 显存，vLLM 显著降低碎片化 |
| **Continuous Batching（连续批处理）** | 新请求可以随时加入批次，不必等凑够一批再处理，GPU 利用率更高。据称可实现约 23 倍吞吐提升 |
| **Chunked Prefill** | 将长输入的预填充阶段分块处理，提高效率 |
| **Speculative Decoding** | 推测解码，加速生成 |
| **优化 CUDA 内核** | 集成 FlashAttention、FlashInfer 等，加速注意力计算 |
| **多种量化** | 支持 GPTQ、AWQ、INT4、INT8、FP8，在显存和精度之间灵活权衡 |

#### 易用性

- **OpenAI 兼容 API**：部署后可直接用 OpenAI 的 API 格式调用
- **流式输出**：支持流式返回生成结果
- **多 LoRA 支持**：可同时加载多个 LoRA 适配器
- **Prefix Caching**：对相同前缀的请求复用缓存，节省计算
- **多硬件支持**：NVIDIA GPU、AMD CPU/GPU、Intel CPU/GPU、Arm、TPU 等

#### 适合谁用？

- 需要**高并发、高吞吐**的生产环境
- 有 GPU 服务器、想自建推理服务的团队
- 需要量化、多模型、分布式推理等高级特性的场景

#### 性能参考

根据官方和社区数据，vLLM 相比 HuggingFace Transformers 可实现约 14–24 倍吞吐提升，相比 HuggingFace TGI 约 2.2–3.5 倍。

### 7.3.2 Ollama：最简单的本地部署方案

> 原文：[Ollama 文档](https://github.com/ollama/ollama/blob/main/docs/README.md)（英文）

Ollama 是在**本地**运行开源 LLM 最省心的工具之一，可以理解为"模型界的 Docker"——帮你下载、管理、运行各种开源模型，无需手动配置 GPU 驱动、模型格式、推理引擎等。

#### 核心特点

- **一行命令跑模型**：`ollama run qwen2.5` 即可
- **自动下载**：首次运行会自动拉取模型
- **OpenAI 兼容**：本地提供 API，可用 OpenAI 客户端直接调用
- **跨平台**：支持 macOS、Linux、Windows

#### 基本使用

```bash
# 安装（macOS / Linux）
curl -fsSL https://ollama.com/install.sh | sh

# 运行模型
ollama run qwen2.5    # 运行 Qwen 2.5
ollama run llama3.1   # 运行 Llama 3.1

# 常用命令
ollama list           # 查看已下载的模型
ollama pull qwen2.5   # 只下载不运行
ollama rm llama3.1    # 删除模型
ollama ps             # 查看正在运行的模型
```

#### 用代码调用

Ollama 默认在 `http://localhost:11434` 提供 API，可用 OpenAI 库调用：

```python
from openai import OpenAI

client = OpenAI(base_url='http://localhost:11434/v1', api_key='ollama')
response = client.chat.completions.create(
    model='qwen2.5',
    messages=[{'role': 'user', 'content': '用一句话解释什么是大语言模型'}]
)
print(response.choices[0].message.content)
```

#### vLLM 和 Ollama 怎么选？

| 维度 | vLLM | Ollama |
|------|------|--------|
| 定位 | 生产级高性能推理 | 本地开发、学习、轻量使用 |
| 性能 | 高吞吐、高并发 | 满足单机、小规模使用 |
| 配置复杂度 | 需要一定配置 | 几乎零配置 |
| 适用场景 | 服务器部署、API 服务 | 个人电脑、快速实验、学习 |

## 7.4 多模态与前沿方向

### 7.4.1 OpenAI Vision 文档：多模态 API 使用指南

> 原文：[OpenAI Vision 文档](https://platform.openai.com/docs/guides/vision)（英文）

OpenAI 的 Vision 文档介绍了如何用 API 处理**图片输入**和**图片生成**，是构建多模态应用（能"看图"的 AI）的入门指南。

#### 多模态能力概览

| API | 能力 |
|-----|------|
| **Responses API / Chat Completions API** | 把图片作为输入，让模型分析图片并生成文字或音频 |
| **Images API** | 根据文字描述生成图片，或基于图片进行编辑 |
| **GPT Image** | 原生多模态模型，既能理解图文，也能生成图片，对指令和上下文的理解更好 |

#### 图片输入的三种方式

1. **URL**：传入图片的完整 URL
2. **Base64**：将图片编码为 Base64 数据 URL
3. **文件 ID**：通过 Files API 上传后使用文件 ID

单次请求可包含多张图片，但会按 Token 计费。

#### 图片输入要求

- **格式**：PNG、JPEG、WEBP、非动图 GIF
- **大小**：单次请求总负载不超过 50 MB，最多 500 张图片
- **内容**：无水印、无 Logo，清晰可辨，非 NSFW

#### 细节级别（detail 参数）

- `low`：低分辨率（约 512×512），固定约 85 Token，适合只关心大致形状、颜色的场景
- `high`：高分辨率，模型能看清更多细节，Token 消耗更高
- `auto`：由模型自动选择（默认）

#### 已知限制

- **验证码**：出于安全原因，不支持解析验证码
- **计数**：对物体数量可能给出近似值
- **空间推理**：对需要精确定位（如棋盘局面）的任务表现一般
- **图表**：对复杂图表、不同线型（实线、虚线等）的理解有限
- **医学影像**：不适合解读 CT 等专业医学影像，不能用于医疗建议

#### 费用

图片按 Token 计费，具体取决于尺寸和 `detail` 设置。可在 OpenAI 定价页使用 Vision 计算器估算。

#### 典型应用

- 看图回答问题（"这张图里有什么？"）
- 图片描述、字幕生成
- 从截图或文档图片中提取信息（OCR）
- 根据文字描述生成或编辑图片

### 7.4.2 CrewAI：多 Agent 协作框架

> 原文：[CrewAI 文档](https://docs.crewai.com/)（英文）

CrewAI 是一个用于构建**多 Agent 协作系统**的框架，强调"生产就绪"——内置记忆、知识库、护栏、可观测性等能力，方便从原型快速推进到生产。

#### 核心概念

- **Agent（智能体）**：有角色、目标、背景故事，可配备工具、记忆和结构化输出
- **Crew（团队）**：多个 Agent 组成一个团队，共同完成任务
- **Task（任务）**：定义要做什么、期望输出、由哪个 Agent 执行
- **Flow（流程）**：编排 start/listen/router 等步骤，管理状态，支持长时间运行的工作流

#### 快速上手示例

文档中的 Quickstart 演示了一个"研究 + 报告"的 Crew：

1. **研究员 Agent**：用 Serper 等工具搜索最新 AI 动态
2. **报告分析师 Agent**：根据研究结果撰写结构化报告
3. **流程**：顺序执行（research → reporting），最终输出 Markdown 报告

#### 项目结构

```
crewai create crew latest-ai-development  # 创建项目
# 生成 agents.yaml、tasks.yaml、crew.py 等
```

- `agents.yaml`：定义 Agent 的角色、目标、背景
- `tasks.yaml`：定义任务描述、期望输出、负责的 Agent
- `crew.py`：用 Python 组装 Agent 和 Task，定义执行流程（如 `Process.sequential`）

#### 企业能力

- **部署**：通过 CrewAI AMP 控制台部署到生产
- **触发器**：连接 Gmail、Slack、Salesforce 等，用事件触发 Crew
- **权限**：团队管理、RBAC（基于角色的访问控制）

#### 适合谁用？

- 需要多个 AI 角色分工协作的场景（如研究 + 写作、分析 + 总结）
- 希望有清晰项目结构、便于维护的团队
- 需要从开发环境平滑过渡到生产部署的项目


**关键认知**：

1. **微调是提示词和 RAG 的下一步**。当提示词和 RAG 无法满足需求时，用自己数据微调模型，可以让小模型在特定任务上媲美大模型，同时保护数据隐私、降低 Token 成本。

2. **评估是优化的前提**。没有可量化的评估，就不知道改得对不对。Ragas 专注 RAG，LMSYS Arena 专注通用对话，两者互补。

3. **部署选型看场景**。学习和小规模使用用 Ollama；生产环境、高并发用 vLLM。两者都支持 OpenAI 兼容 API，迁移成本低。

4. **多模态和多 Agent 是重要方向**。Vision API 让应用能"看图"，CrewAI 让多个 Agent 分工协作，适合复杂工作流。
