## 知识模块

### 7.1 模型微调（Fine-tuning）

- **是什么**：微调就像"给通才做专项培训"。LLM 是通用的，但你可以用自己的数据对它进行额外训练，让它在特定领域表现更好。就像一个全科医生经过专科培训后，在心脏外科方面会比通才强得多。

- **为什么重要**：当提示词工程和 RAG 都无法满足需求时（比如需要模型学会特定的输出风格、掌握专业领域知识、或大幅降低推理成本），微调是下一步选择。

- **核心概念**：
  - **全量微调 vs LoRA**：全量微调修改所有参数，效果好但需要大量 GPU；LoRA（Low-Rank Adaptation）只修改少量参数，效率高，是目前最主流的微调方法。
  - **训练数据准备**：微调数据通常是"输入-输出"对的格式。数据质量比数量更重要——100 条高质量数据可能比 10000 条低质量数据效果好。
  - **SFT（Supervised Fine-Tuning）**：监督微调，用标注好的数据训练模型按照期望的方式回答。
  - **RLHF / DPO**：通过人类反馈或偏好数据进一步对齐模型行为，让模型更符合人类期望。
  - **评估与防止过拟合**：微调后要评估模型在测试集上的表现，避免模型只记住了训练数据而失去泛化能力。

- **动手练习**：
  1. 用 OpenAI 的 Fine-tuning API 微调一个 GPT-4o-mini，让它学会特定的输出格式
  2. 用 Unsloth 或 LLaMA-Factory 在本地对 Qwen2.5-7B 做 LoRA 微调
  3. 对比微调前后模型在目标任务上的表现差异

- **推荐资源**：
  - 📖 必读：[OpenAI Fine-tuning 文档](https://platform.openai.com/docs/guides/fine-tuning) — 最简单的微调入门（英文）
  - 📖 补充：[Unsloth](https://github.com/unslothai/unsloth) — 高效本地微调工具，速度快、显存省
  - 🛠️ 工具：OpenAI Fine-tuning API、Unsloth、LLaMA-Factory、Hugging Face Transformers

- **检验标准**：
  - 什么时候该用微调而不是提示词工程或 RAG？
  - LoRA 的基本原理是什么？为什么比全量微调高效？
  - 微调数据应该如何准备？

### 7.2 模型评估与基准测试

- **是什么**：模型评估就像"考试"——用标准化的题目测试模型的各项能力，得出可量化的分数。这样你才能客观地比较不同模型、不同版本、不同配置的效果。

- **为什么重要**：在实际项目中，你经常需要回答"该用哪个模型？""微调后效果好了还是差了？""新版本的提示词比旧版本好吗？"这些问题都需要可靠的评估方法。

- **核心概念**：
  - **通用基准测试**：MMLU（多任务语言理解）、HumanEval（代码生成）、GSM8K（数学推理）等标准化测试集。
  - **领域评估**：针对特定业务场景设计的评估方法。比如客服场景评估回答的准确性和礼貌程度。
  - **LLM-as-Judge**：用一个强模型（如 GPT-4o）来评估另一个模型的输出质量。成本低、可扩展，但需要设计好评估提示词。
  - **人工评估**：最可靠但最昂贵的方法。通常用于验证自动评估的准确性。
  - **RAG 评估指标**：Faithfulness（忠实度）、Relevancy（相关性）、Context Precision（上下文精度）等。

- **动手练习**：
  1. 用 Ragas 框架评估你在阶段 4 构建的 RAG 系统
  2. 设计一个 LLM-as-Judge 评估流程：定义评估维度、编写评估提示词、对 50 个样本打分
  3. 对比人工评估和 LLM-as-Judge 的一致性

- **推荐资源**：
  - 📖 必读：[Ragas 文档](https://docs.ragas.io/) — RAG 评估框架（英文）
  - 📖 补充：[LMSYS Chatbot Arena](https://chat.lmsys.org/) — 基于人类偏好的模型评估平台
  - 🛠️ 工具：Ragas、OpenAI Evals、LangSmith Evaluation

- **检验标准**：
  - 常见的 LLM 评估基准有哪些？各测试什么能力？
  - LLM-as-Judge 的优缺点是什么？
  - 如何为自己的业务场景设计评估方案？

### 7.3 模型部署与推理优化

- **是什么**：模型部署就像"把实验室的原型机搬到工厂量产"——让模型能在生产环境中稳定、高效、低成本地运行，服务真实用户。

- **为什么重要**：本地跑个 demo 和服务百万用户是完全不同的挑战。你需要考虑并发处理、推理速度、GPU 利用率、成本优化等工程问题。

- **核心概念**：
  - **推理框架**：vLLM（高性能推理引擎，支持 PagedAttention）、TGI（Hugging Face 的推理服务）、Ollama（简单易用的本地部署方案）。
  - **量化（Quantization）**：把模型参数从高精度（FP16）压缩到低精度（INT8/INT4），大幅减少显存占用和推理时间，代价是轻微的质量损失。
  - **批处理（Batching）**：把多个请求合并处理，提高 GPU 利用率。连续批处理（Continuous Batching）是目前最先进的方案。
  - **KV Cache**：缓存注意力计算的中间结果，避免重复计算，加速生成。
  - **模型即服务（Model-as-a-Service）**：通过 API 提供模型推理服务，可以用 Docker + FastAPI + vLLM 快速搭建。

- **动手练习**：
  1. 用 vLLM 部署一个 Qwen2.5-7B 模型，提供 OpenAI 兼容的 API
  2. 对比 FP16 和 INT4 量化版本的推理速度和输出质量
  3. 用 Apache Bench 或 wrk 对你部署的模型做压力测试

- **推荐资源**：
  - 📖 必读：[vLLM 官方文档](https://docs.vllm.ai/) — 最流行的高性能推理引擎（英文）
  - 📖 补充：[Ollama 文档](https://github.com/ollama/ollama/blob/main/docs/README.md) — 最简单的本地部署方案
  - 🛠️ 工具：vLLM、Ollama、Docker、NVIDIA GPU

- **检验标准**：
  - vLLM 和 Ollama 各适合什么场景？
  - 量化是什么？INT4 量化对模型质量有多大影响？
  - 如何评估模型部署的性能（QPS、延迟、吞吐量）？

### 7.4 多模态与前沿方向

- **是什么**：多模态就像给 AI 装上了"眼睛和耳朵"——不再只能处理文字，还能理解图片、音频、视频。前沿方向则是 LLM 领域正在快速发展的新技术。

- **为什么重要**：AI 应用正从纯文本走向多模态。理解图片、生成图片、语音交互等能力正在成为 AI 产品的标配。了解前沿方向能帮你把握技术趋势，提前布局。

- **核心概念**：
  - **视觉语言模型（VLM）**：能同时理解图片和文字的模型，如 GPT-4o、Claude 3.5 Sonnet。可以看图回答问题、描述图片内容、从截图提取信息。
  - **图像生成**：从文字描述生成图片，如 DALL-E 3、Midjourney、Stable Diffusion。
  - **语音交互**：语音转文字（Whisper）+ LLM + 文字转语音（TTS），实现语音对话。
  - **AI 编程助手**：Cursor、GitHub Copilot 等工具背后的技术——代码补全、代码生成、代码解释。
  - **多 Agent 系统**：多个 Agent 分工协作，处理更复杂的任务。如 CrewAI、AutoGen。

- **动手练习**：
  1. 用 GPT-4o 的 Vision API 实现一个"看图说话"功能：上传图片，模型描述内容
  2. 用 Whisper API 实现语音转文字，再接入 LLM 生成回答
  3. 用 CrewAI 搭建一个 2 Agent 协作系统（如：研究员 Agent + 写作 Agent）

- **推荐资源**：
  - 📖 必读：[OpenAI Vision 文档](https://platform.openai.com/docs/guides/vision) — 多模态 API 使用指南（英文）
  - 📖 补充：[CrewAI 文档](https://docs.crewai.com/) — 多 Agent 协作框架（英文）
  - 🛠️ 工具：OpenAI Vision API、Whisper、CrewAI、Stable Diffusion

- **检验标准**：
  - 多模态模型能处理哪些类型的输入？
  - 语音交互的技术栈通常包含哪些组件？
  - 多 Agent 系统相比单 Agent 有什么优势和挑战？

## 阶段项目：选择一个方向深入

根据你的兴趣，从以下项目中选择一个完成：

### 选项 A：微调专家

用自己收集的数据微调一个开源模型，使其在特定任务上超越通用模型。
- 收集/整理 200+ 条高质量训练数据
- 用 LoRA 微调 Qwen2.5-7B
- 构建评估数据集，量化对比微调前后的效果
- 预计耗时：10-12 小时

### 选项 B：部署工程师

搭建一个完整的模型推理服务，能处理并发请求。
- 用 vLLM 部署量化后的开源模型
- 实现 OpenAI 兼容的 API
- 添加限流、日志、健康检查
- 用 Docker Compose 编排，一键启动
- 预计耗时：8-10 小时

### 选项 C：多模态应用

开发一个多模态 AI 应用。
- 支持图片输入（图片理解、OCR）
- 支持语音输入（Whisper 转文字）
- 结合 RAG 实现基于图文文档的问答
- 预计耗时：10-12 小时
