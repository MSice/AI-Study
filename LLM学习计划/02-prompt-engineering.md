**学习目标**：掌握提示词工程的核心技巧，能针对不同任务设计高质量的提示词，显著提升 LLM 输出质量。

**前置要求**：完成阶段 1（LLM 基础认知）

---

## 知识模块

### 2.1 提示词工程基础

- **是什么**：提示词工程就像学会"跟 AI 说话的艺术"。同样一件事，你怎么说，决定了 AI 怎么做。就像给实习生布置任务——说得越清楚，结果越靠谱。

- **为什么重要**：提示词是你和 LLM 之间唯一的沟通界面。好的提示词能让普通模型发挥出超常水平，差的提示词会让顶级模型也输出垃圾。

- **核心概念**：
  - **指令（Instruction）**：明确告诉模型要做什么。"请翻译以下文本" 比 "帮我看看这段话" 效果好得多。
  - **上下文（Context）**：给模型提供背景信息。就像给医生描述症状——信息越完整，诊断越准确。
  - **输入数据（Input Data）**：需要模型处理的具体内容，用分隔符（如 ```、---、XML 标签）和指令区分开。
  - **输出格式（Output Format）**：明确告诉模型你要什么格式。"请用 JSON 格式输出" 比 "给我结果" 好得多。

- **动手练习**：
  1. 选一个任务（如"给一篇技术文章写摘要"），先用随意的提示词试一次，再用结构化提示词试一次，对比效果
  2. 练习用 XML 标签（`<context>`, `<input>`, `<output>`）组织提示词结构
  3. 尝试让 LLM 以 JSON、Markdown 表格、YAML 等不同格式输出

- **推荐资源**：
  - 📖 必读：[Anthropic Prompt Engineering Guide](https://docs.anthropic.com/en/docs/build-with-claude/prompt-engineering/overview) — Anthropic 官方提示词工程指南，系统全面（英文）
  - 🎬 推荐：[OpenAI Prompt Engineering Best Practices](https://platform.openai.com/docs/guides/prompt-engineering) — OpenAI 官方最佳实践（英文）
  - 🛠️ 工具：ChatGPT / Claude 对话界面、Cursor IDE

- **检验标准**：
  - 一个好的提示词应该包含哪几个要素？
  - 为什么要用分隔符区分指令和输入数据？
  - 如何让 LLM 稳定输出 JSON 格式？

### 2.2 高级提示技巧

- **是什么**：如果基础提示词是"跟 AI 说人话"，高级技巧就是"教 AI 怎么思考"。就像好老师不只告诉学生答案，还教他们解题思路。

- **为什么重要**：面对复杂任务（多步推理、代码生成、数据分析），基础提示词往往不够用。高级技巧能大幅提升模型在复杂任务上的表现。

- **核心概念**：
  - **Few-shot Learning（少样本学习）**：在提示词中给几个示例，让模型"照葫芦画瓢"。就像给新员工看几个范例文档。
  - **Chain of Thought / CoT（思维链）**：让模型"一步一步想"，而不是直接给答案。加一句"请一步步分析"就能显著提升推理准确率。
  - **角色扮演（Role Playing）**：给模型设定一个专家身份。"你是一名资深后端工程师"比不设角色效果好。
  - **自我反思（Self-Reflection）**：让模型检查自己的输出。"请检查你的回答是否有逻辑错误" 能减少低级错误。
  - **分解任务（Task Decomposition）**：把复杂任务拆成多个简单步骤，逐步完成。

- **动手练习**：
  1. 用 Few-shot 技巧让 LLM 学会一种特定的文本分类规则（如情感分析：给 3 个例子，然后分类新文本）
  2. 对比同一道数学题，直接问 vs 加 "请一步步推理" 的效果差异
  3. 设计一个多步骤提示词：先让模型分析需求 → 设计方案 → 写代码 → 自我审查

- **推荐资源**：
  - 📖 必读：[Prompt Engineering Guide](https://www.promptingguide.ai/zh) — 社区维护的提示词工程百科全书，有中文版
  - 🎬 推荐：[Chain-of-Thought Prompting 原始论文](https://arxiv.org/abs/2201.11903) — 理解 CoT 的学术来源（英文）
  - 🛠️ 工具：LangSmith（提示词调试和追踪）

- **检验标准**：
  - Few-shot 和 Zero-shot 的区别是什么？各适合什么场景？
  - Chain of Thought 为什么能提升推理能力？
  - 如何设计一个包含自我反思的提示词？

### 2.3 提示词模板设计

- **是什么**：提示词模板就像"万能公式"——针对常见任务类型，提前设计好可复用的提示词框架，用的时候只需要填空。就像公司的合同模板，每次只改关键信息。

- **为什么重要**：在实际开发中，你不可能每次都从零写提示词。好的模板能保证输出质量的一致性，也方便团队协作和维护。

- **核心概念**：
  - **模板变量**：模板中需要动态替换的部分，用占位符标记（如 `{{topic}}`、`{input_text}`）。
  - **模板复用**：同一个模板适配不同输入，保证输出风格和质量一致。
  - **模板版本管理**：像管理代码一样管理提示词，记录每次修改和效果变化。
  - **提示词链（Prompt Chain）**：多个模板串联，前一个的输出作为后一个的输入，处理复杂的多步骤任务。

- **动手练习**：
  1. 设计 3 个实用的提示词模板：代码审查模板、技术文档生成模板、Bug 分析模板
  2. 用 Python 的 f-string 或 Jinja2 实现模板变量替换
  3. 设计一个 2 步提示词链：第一步生成大纲，第二步根据大纲生成完整内容

- **推荐资源**：
  - 📖 必读：[LangChain Prompt Templates 文档](https://python.langchain.com/docs/concepts/prompt_templates/) — 工程化管理提示词模板的标准方式（英文）
  - 🛠️ 工具：LangChain、Jinja2 模板引擎

- **检验标准**：
  - 如何设计一个可复用的提示词模板？
  - 提示词链的工作原理是什么？举一个实际应用场景
  - 为什么需要对提示词做版本管理？

## 阶段项目：万能提示词工具箱

**项目描述**：开发一个命令行工具，内置 5 个实用的提示词模板（代码审查、文档生成、翻译润色、SQL 生成、Bug 分析），用户选择模板后填入参数，自动调用 LLM API 并输出结果。

**技术要求**：
- Python CLI 工具，使用 `click` 或 `argparse`
- 提示词模板用 YAML/JSON 文件管理，支持变量替换
- 支持选择不同的 LLM 后端（OpenAI / Ollama）
- 支持流式输出

**预期产出**：
- `prompt_toolkit/` — 项目目录
  - `templates/` — 提示词模板文件
  - `cli.py` — 命令行入口
  - `llm_client.py` — LLM 调用封装
- `README.md` — 使用说明

**预计耗时**：4-5 小时
