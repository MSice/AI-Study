## 知识模块

### 4.1 RAG 原理与架构

- **是什么**：RAG（Retrieval-Augmented Generation，检索增强生成）就像"开卷考试"——LLM 本身的知识有限且可能过时，RAG 让它在回答问题前先去"翻书"（检索相关文档），然后基于找到的内容组织答案。这样既利用了 LLM 的语言能力，又保证了答案基于真实数据。

- **为什么重要**：LLM 的训练数据有截止日期，且不包含你的私有数据（公司文档、产品手册等）。RAG 是让 LLM "懂你的业务"的最实用方案，不需要微调模型，成本低、见效快。

- **核心概念**：
  - **Embedding（向量嵌入）**：给每段文本生成一张"多维身份证"——一组数字（向量）。语义相似的文本，向量也相近。这是 RAG 能"按意思搜索"的基础。
  - **向量数据库（Vector Database）**：专门存储和检索向量的数据库。你存入文档的向量，查询时用问题的向量去找最相似的文档。就像一个能按"意思"而非"关键词"搜索的图书馆。
  - **文档分块（Chunking）**：把长文档切成小段。太长模型处理不了，太短丢失上下文。分块策略直接影响检索质量。
  - **检索（Retrieval）**：根据用户问题找到最相关的文档片段。常用方法有向量相似度搜索、关键词搜索、混合搜索。
  - **生成（Generation）**：把检索到的文档片段和用户问题一起喂给 LLM，让它基于这些"参考资料"生成回答。

- **动手练习**：
  1. 用 OpenAI 的 Embedding API 把 5 段不同主题的文本转为向量，计算它们之间的余弦相似度
  2. 画出 RAG 的完整数据流图：文档 → 分块 → 向量化 → 存储 → 查询 → 检索 → 生成
  3. 思考：为什么不直接把所有文档塞进 LLM 的上下文窗口？（提示：成本、速度、窗口限制）

- **推荐资源**：
  - 📖 必读：[RAG from Scratch (LangChain)](https://github.com/langchain-ai/rag-from-scratch) — 从零实现 RAG 的系列教程（英文）
  - 🎬 推荐：[What is Retrieval-Augmented Generation (RAG)?](https://www.youtube.com/watch?v=T-D1OfcDW1M) — IBM 的 RAG 科普视频（英文）
  - 🛠️ 工具：OpenAI Embedding API、Pinecone / Qdrant / ChromaDB

### 4.2 文档处理与分块策略

- **是什么**：文档处理就像"备菜"——在正式"做菜"（检索和生成）之前，你需要把各种格式的原料（PDF、Word、网页、Markdown）清洗干净、切成合适大小的块。切得好不好，直接决定最终"菜品"的质量。

- **为什么重要**：分块策略是 RAG 系统中最容易被忽视但影响最大的环节。分块太大，检索不精准；分块太小，丢失上下文；分块方式不对，可能把一个完整概念切断。

- **核心概念**：
  - **文档加载（Document Loading）**：把 PDF、Word、HTML、Markdown 等格式的文件读取为纯文本。不同格式需要不同的解析器。
  - **固定大小分块**：按字符数或 Token 数切分，简单粗暴但可能切断语义。
  - **语义分块**：按段落、标题、章节等语义边界切分，保持内容完整性。
  - **重叠分块（Overlap）**：相邻块之间有一部分重叠内容，避免关键信息恰好在切分点被截断。
  - **元数据（Metadata）**：给每个块附加额外信息（来源文件、页码、标题层级等），方便后续过滤和溯源。

- **动手练习**：
  1. 用 `langchain` 的文档加载器读取一个 PDF 文件和一个 Markdown 文件
  2. 分别用固定大小分块（500 字符，100 字符重叠）和按标题分块两种策略处理同一个文档，对比结果
  3. 给分块添加元数据（文件名、块序号），存入一个列表并打印

- **推荐资源**：
  - 📖 必读：[Chunking Strategies for LLM Applications](https://www.pinecone.io/learn/chunking-strategies/) — Pinecone 的分块策略详解（英文）
  - 🛠️ 工具：LangChain Document Loaders、Unstructured.io

- **检验标准**：
  - 为什么分块策略对 RAG 质量影响很大？
  - 固定大小分块和语义分块各有什么优缺点？
  - Overlap（重叠）的作用是什么？

### 4.3 向量数据库与检索

- **是什么**：向量数据库就像一个"按意思分类的超级图书馆"。传统数据库按关键词搜索（搜"苹果"只能找到包含"苹果"的文档），向量数据库按语义搜索（搜"水果"也能找到关于"苹果"的文档）。

- **为什么重要**：向量数据库是 RAG 系统的核心基础设施。选对数据库、用对检索策略，直接决定了用户问题能否找到正确的答案。

- **核心概念**：
  - **主流向量数据库对比**：ChromaDB（轻量本地，适合开发测试）、Pinecone（云托管，开箱即用）、Qdrant（高性能，支持过滤）、Weaviate（功能丰富）、FAISS（Facebook 开源，纯向量搜索库）。
  - **相似度度量**：余弦相似度（最常用）、欧氏距离、点积。不同度量方式适合不同场景。
  - **混合搜索（Hybrid Search）**：结合向量搜索（语义匹配）和关键词搜索（精确匹配），取两者之长。
  - **Top-K 与阈值**：检索时返回最相似的 K 个结果，或设置相似度阈值过滤低质量结果。
  - **重排序（Reranking）**：用专门的模型对初步检索结果重新排序，提升最终结果的相关性。

- **动手练习**：
  1. 用 ChromaDB 在本地创建一个向量集合，存入 20 段文本，实现语义搜索
  2. 对比纯向量搜索和混合搜索的效果差异
  3. 尝试不同的 Top-K 值（3、5、10），观察对最终回答质量的影响

- **推荐资源**：
  - 📖 必读：[ChromaDB 官方文档](https://docs.trychroma.com/) — 最简单的向量数据库入门（英文）
  - 📖 补充：[Qdrant 官方文档](https://qdrant.tech/documentation/) — 生产级向量数据库（英文）
  - 🛠️ 工具：ChromaDB、Qdrant、FAISS

- **检验标准**：
  - 向量数据库和传统数据库的核心区别是什么？
  - 什么是混合搜索？为什么比纯向量搜索效果好？
  - 如何选择适合自己场景的向量数据库？

### 4.4 RAG 优化技巧

- **是什么**：基础 RAG 就像"及格线"——能用但不够好。优化 RAG 就是从 60 分提升到 90 分的过程，涉及检索质量、生成质量、系统效率等多个维度。

- **为什么重要**：实际项目中，基础 RAG 的效果往往不能满足需求（检索不准、回答不全、幻觉仍在）。掌握优化技巧是 RAG 从 demo 走向生产的关键。

- **核心概念**：
  - **查询改写（Query Rewriting）**：用户的问题可能模糊或不完整，先用 LLM 改写/扩展查询，再去检索。
  - **HyDE（假设文档嵌入）**：先让 LLM 生成一个"假设的理想答案"，用这个答案去检索，往往比用原始问题检索效果更好。
  - **多路召回（Multi-Query Retrieval）**：把一个问题改写成多个不同角度的查询，分别检索后合并结果，提升召回率。
  - **上下文压缩（Context Compression）**：检索到的文档可能包含大量无关内容，用 LLM 提取其中与问题相关的部分，减少噪音。
  - **引用溯源（Citation）**：在回答中标注信息来源（哪个文档、哪一页），增强可信度。

- **动手练习**：
  1. 实现查询改写：用户输入 → LLM 改写为更精确的查询 → 检索 → 生成
  2. 实现 HyDE：用户输入 → LLM 生成假设答案 → 用假设答案检索 → 生成最终答案
  3. 对比基础 RAG 和加了查询改写后的 RAG，用 5 个测试问题评估效果差异

- **推荐资源**：
  - 📖 必读：[Advanced RAG Techniques](https://www.llamaindex.ai/blog/a-cheat-sheet-and-some-recipes-for-building-advanced-rag) — LlamaIndex 的高级 RAG 技巧汇总（英文）
  - 🛠️ 工具：LangChain、LlamaIndex

- **检验标准**：
  - 基础 RAG 常见的问题有哪些？
  - 查询改写和 HyDE 分别解决什么问题？
  - 如何评估 RAG 系统的效果？

## 阶段项目：私有知识库问答系统

**项目描述**：构建一个基于本地文档的智能问答系统。用户可以上传文档（PDF/Markdown/TXT），系统自动处理并建立知识库，然后用户可以用自然语言提问，系统基于文档内容回答并标注来源。

**技术要求**：
- 支持 PDF、Markdown、TXT 三种文档格式
- 使用 ChromaDB 或 Qdrant 作为向量数据库
- 实现至少一种 RAG 优化技巧（查询改写 / HyDE / 重排序）
- 回答中包含引用来源（文件名 + 相关段落）
- 提供 CLI 或简单的 Web 界面（可用 Streamlit / Gradio）

**预期产出**：
- `rag_qa/` — 项目目录
  - `ingest.py` — 文档处理和入库
  - `retriever.py` — 检索逻辑
  - `generator.py` — 生成逻辑
  - `app.py` — 用户界面
- `README.md` — 使用说明和架构图

**预计耗时**：8-10 小时
