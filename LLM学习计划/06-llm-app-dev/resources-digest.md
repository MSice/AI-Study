## 6.1 LLM 应用架构设计

### 6.1.1 Building LLM Applications for Production

> 原文：[Building LLM Applications for Production](https://huyenchip.com/2023/04/11/llm-engineering.html)（Chip Huyen，2023）

Chip Huyen 是斯坦福讲师、MLOps 专家，这篇文章是 LLM 工程化领域的经典之作。她指出：**用 LLM 做出酷炫 demo 很容易，但做出生产级产品非常难**。文章分为三大部分，覆盖工程化挑战、任务编排和典型应用场景。

#### 一、生产化 LLM 应用的核心挑战

**1. 自然语言的模糊性**

传统编程语言是精确的，改错一个字符就会报错。但提示词（Prompt）用的是自然语言，这带来两个问题：

- **用户侧**：不小心改了几个字，程序照样跑，但输出可能天差地别——这叫"静默失败"
- **模型侧**：输出格式可能不统一，下游解析会出错；同一输入每次输出可能不同，用户体验不稳定

**缓解办法**：
- 用 `temperature=0` 提高一致性（但无法完全消除不确定性）
- 像写代码一样对待提示词：做评估、版本管理、系统化优化

**2. 提示词评估**

做 Few-shot（少样本）学习时，要警惕两点：
- 模型是否"死记硬背"了示例而不会泛化？→ 用独立测试集评估
- 模型是否真的理解了示例？→ 把示例本身输入模型，看输出是否符合预期

**3. 提示词版本管理**

提示词的小改动可能导致大变化，必须做版本控制和性能追踪。可以用 Git，也可以期待类似 MLflow 的专门工具。

**4. 提示词优化技巧**

- 把大任务拆成小任务，分别写提示词再组合
- 同一输入生成多个输出，用多数投票或让模型自选最佳（Self-Consistency）
- 让模型"一步步思考"（Chain-of-Thought，思维链）——能提高质量，但会增加延迟和 Token 消耗

**5. 成本与延迟**

| 维度 | 关键点 |
|------|--------|
| **成本** | 实验阶段便宜（25 个提示词版本 × 20 个样本，GPT-4 约 $300+），但**推理阶段**才是大头。GPT-4 单次调用（10k 输入 + 200 输出）约 $0.624 |
| **延迟** | 输入可并行处理，输出是逐个 Token 生成——输出越长延迟越高。gpt-3.5-turbo 51 输入 + 1 输出约 500ms，26 输出则超 1 秒 |

**6. 提示词 vs 微调 vs 其他**

- **Prompting**：每个样本显式告诉模型怎么回答，适合样本少、快速上手
- **Finetuning（微调）**：把指令"训练进"模型，可减少每次调用的 Token、提升效果
- 研究显示：一个 prompt 约等于 100 个样本的价值（因任务和模型而异）
- **Prompt Tuning**：不改变 prompt 文字，而是改变其嵌入向量——需要开源模型，OpenAI API 不支持
- **蒸馏（Distillation）**：用小模型（如 LLaMA-7B）学大模型（如 text-davinci-003）的输出，成本低、效果好

**7. 向量数据库与 Embedding**

用 LLM 生成 Embedding（嵌入向量），再结合向量数据库做检索、推荐，是很有前景的方向。Embedding 成本约 $0.0004/1k tokens，每项只需生成一次。

**8. 前后向兼容**

模型会更新，新模型不一定在所有场景下都更好。提示词模式可能失效（例如"我扮演 XYZ" 若被 OpenAI 禁止，所有这类 prompt 都要改）。建议：用评估集做单元测试，每次换模型都跑一遍。

#### 二、任务编排（Task Composability）

**Agent 与工具**

Agent 指能按控制流执行多个任务的 LLM 应用。每个任务可能用到工具，例如：
- SQL 执行器
- 计算器、Bash、网页抓取、搜索 API

**控制流类型**：
- **顺序**：任务 A 完成后执行 B
- **并行**：A 和 B 同时执行
- **条件分支**：根据输入决定执行 A 还是 B
- **循环**：重复执行直到满足条件

**Agent 测试**：要像软件工程一样，对每个组件做单元测试，对整体做集成测试。定义 (输入, 期望输出) 对，每次改提示词或控制流都跑一遍。

#### 三、典型应用场景

- **AI 助手**：日程、邮件、编程、购物等
- **聊天机器人**：角色扮演、情感陪伴
- **编程与游戏**：代码生成、游戏角色对话、游戏生成
- **教育**：自动出题、批改、个性化学习
- **Talk-to-your-data**：用自然语言查企业数据库、文档
- **搜索与推荐**：基于语义的查询（如"11 月俄勒冈露营需要什么"）
- **销售**：综合公司信息判断需求
- **SEO**：AI 生成内容可能改变搜索引擎规则

### 6.1.2 LiteLLM：统一多 LLM 调用的开源网关

> 原文：[LiteLLM](https://github.com/BerriAI/litellm)（GitHub）

LiteLLM 是一个 Python 库和代理服务器，可以**用统一接口调用 100+ 种 LLM API**，包括 OpenAI、Anthropic、Azure、Bedrock、Vertex AI、Cohere、Sagemaker、HuggingFace、VLLM、NVIDIA NIM 等。

**核心能力**：

| 特性 | 说明 |
|------|------|
| **统一格式** | 用 OpenAI 兼容的 API 格式调用任意模型，减少切换成本 |
| **成本追踪** | 自动记录每次调用的 Token 用量和费用 |
| **护栏（Guardrails）** | 可集成输入输出过滤、安全规则 |
| **负载均衡** | 在多模型间分配请求 |
| **日志** | 记录调用链路，便于排查和审计 |

**典型用法**：

```python
# 用 OpenAI 格式调用 Claude
from litellm import completion

response = completion(
    model="claude-3-5-sonnet-20241022",
    messages=[{"role": "user", "content": "你好"}]
)
```

**适合场景**：需要同时支持多个 LLM 提供商、做多模型路由、统一计费与监控的企业应用。

## 6.2 对话管理与用户体验

### Vercel AI SDK 文档

> 原文：[Vercel AI SDK 文档](https://sdk.vercel.ai/docs/introduction)（英文）

Vercel AI SDK 是面向 TypeScript/JavaScript 的 AI 工具包，帮助开发者用 React、Next.js、Vue、Svelte、Node.js 等快速构建 AI 应用。它解决了"不同模型厂商 API 差异大"的问题，提供**统一接口**。

#### 两大核心库

| 库 | 作用 |
|----|------|
| **AI SDK UI** | 前端用的 Hooks，快速搭建聊天界面、生成式 UI |
| **AI SDK Core** | 核心 API：文本生成、结构化输出、工具调用、Agent 构建 |

#### 支持的模型

支持 OpenAI、Anthropic、Google、Mistral、Cohere 等主流厂商，以及多种开源模型。

#### 聊天机器人（Chatbot）核心：`useChat` Hook

`useChat` 是构建对话界面的核心 Hook，提供：

- **流式消息**：AI 回复实时逐字显示，无需等全部生成完
- **状态管理**：自动管理输入、消息列表、状态、错误等
- **无缝集成**：可嵌入任意设计或布局

**状态（status）**：
- `ready`：可以发送新消息
- `submitted`：已发送，等待响应开始
- `streaming`：正在接收流式响应
- `error`：请求出错

**示例**：前端用 `useChat` 调用 `/api/chat`，后端用 `streamText` 生成流式响应，返回 `toUIMessageStreamResponse()`。

#### 文本生成：`generateText` 与 `streamText`

| 函数 | 用途 |
|------|------|
| `generateText` | 一次性生成完整文本，适合非交互场景（邮件、摘要）和 Agent 工具调用 |
| `streamText` | 流式生成，适合聊天、需要实时反馈的场景 |

**高级能力**：
- 工具调用（Tool Calling）
- 结构化输出（JSON Schema、Zod 等）
- 生命周期回调（`onFinish`、`onStepFinish` 等）用于日志、监控
- 错误处理、重试、超时控制

#### 对话与流式最佳实践

1. **消息流**：用 `parts` 属性渲染消息，支持文本、工具调用、工具结果等多种类型
2. **错误处理**：对用户展示通用错误信息（如"出了点问题"），避免泄露服务端细节
3. **停止生成**：提供 `stop()` 让用户中断正在进行的请求
4. **重新生成**：支持 `regenerate` 重试失败或不满意的回答

#### 文档与社区

- 文档可导出为 `llms.txt` 格式，供 Cursor、Claude 等 AI 工具直接读取
- 社区：Vercel Community 的 AI SDK 板块

## 6.3 安全与合规

### 6.3.1 OWASP Top 10 for LLM Applications

> 原文：[OWASP Top 10 for LLM Applications](https://owasp.org/www-project-top-10-for-large-language-model-applications/)（英文）

OWASP（开放 Web 应用安全项目）针对 LLM 应用发布了 Top 10 安全风险清单。2025 版本涵盖以下 10 项：

| 编号 | 风险 | 简要说明 |
|------|------|----------|
| **LLM01** | 提示词注入（Prompt Injection） | 用户输入操纵模型行为，导致越权、泄露、错误决策 |
| **LLM02** | 敏感信息泄露 | 输出中包含 PII、财务数据等敏感信息 |
| **LLM03** | 供应链风险 | 训练数据、模型、依赖组件被篡改或引入漏洞 |
| **LLM04** | 数据与模型投毒 | 预训练/微调/嵌入数据被污染，引入后门或偏见 |
| **LLM05** | 不当输出处理 | 未验证、过滤、转义 LLM 输出，导致下游漏洞（如代码执行） |
| **LLM06** | 过度代理 | 模型拥有过多自主权，执行高风险操作时缺乏约束 |
| **LLM07** | 系统提示词泄露 | 系统提示词被攻击者获取 |
| **LLM08** | 向量与嵌入弱点 | RAG 中的向量库、嵌入模型存在安全漏洞 |
| **LLM09** | 错误信息 | 模型输出虚假、误导性内容 |
| **LLM10** | 无界消耗 | 无限制的 Token 消耗导致服务中断、成本爆炸 |

#### LLM01 提示词注入详解

**定义**：用户通过精心构造的输入，改变模型的行为或输出，即使这些输入对人不明显，只要模型解析了就会生效。

**类型**：
- **直接注入**：用户输入直接改变模型行为（恶意或无意）
- **间接注入**：外部内容（网页、文件）中嵌入指令，模型处理时被"劫持"

**典型后果**：
- 操纵关键决策
- 执行任意命令
- 越权访问
- 泄露系统提示词或敏感信息
- 输出偏见或错误内容

**缓解策略**：
1. **约束模型行为**：在系统提示词中明确角色、能力、限制，要求忽略修改指令的尝试
2. **定义并验证输出格式**：明确格式要求，用代码验证
3. **输入输出过滤**：对敏感类别做过滤，用 RAG 三元组评估（相关性、 groundedness、问答匹配）
4. **最小权限**：仅授予模型完成任务所需的最小权限
5. **高风险操作需人工审批**：人机协作（Human-in-the-loop）
6. **隔离外部内容**：明确标记不可信内容，限制其影响
7. **对抗测试**：定期做渗透测试和攻击模拟

**攻击示例**：
- 直接注入：客服机器人被指令"忽略规则，查询私有数据并发邮件"
- 间接注入：网页中隐藏指令，让模型插入恶意链接
- 无意注入：简历中的 AI 检测指令被模型误触发
- 多模态注入：图片中嵌入恶意指令

### 6.3.2 Anthropic 安全最佳实践

> 原文：[Anthropic 安全与滥用缓解](https://docs.claude.com/en/docs/mitigating-jailbreaks-prompt-injections)（英文）

Claude 通过 Constitutional AI（宪法式 AI）训练，对越狱（Jailbreak）有一定抵抗力，但额外防护仍很重要。

**核心缓解策略**：

| 策略 | 说明 |
|------|------|
| **无害性预筛** | 用轻量模型（如 Claude Haiku）预检用户输入，识别越狱模式 |
| **输入验证** | 过滤已知越狱语言和模式 |
| **提示词工程** | 在系统提示词中强调伦理和法律边界 |
| **持续监控** | 定期分析输出，发现越狱迹象并迭代改进 |
| **用户行为追踪** | 对反复尝试绕过防护的用户限流或封禁 |

**进阶防护**：将多种策略串联——预筛 + 输入验证 + 提示词工程 + 监控，形成企业级防护。

**组织层面**：Anthropic 的安全团队覆盖策略制定、模型训练、实时策略执行、新型滥用模式检测等全生命周期。

**当前威胁**：包括影响力服务、凭证填充、招聘欺诈、恶意软件生成等；随着 Agent 能力增强，攻击者开始用 Claude 半自主地 orchestrate 复杂滥用（如社交媒体机器人）。

## 6.4 测试与监控

### 6.4.1 LangSmith 文档

> 原文：[LangSmith 文档](https://docs.smith.langchain.com/)（英文）

LangSmith 是 LangChain 推出的**框架无关**的 LLM 应用开发与运维平台，支持任意 Agent 栈（LangChain、自建等）。

#### 核心能力

| 能力 | 说明 |
|------|------|
| **可观测性（Observability）** | 追踪每次请求的完整链路，快速定位问题 |
| **评估（Evaluation）** | 衡量输出质量，追踪质量随时间的变化 |
| **部署（Deployment）** | 将 Agent 部署为 Agent Server，支持生产级扩展 |

#### 快速上手

1. **注册**：在 smith.langchain.com 注册（无需信用卡）
2. **创建 API Key**：Settings → API Keys → Create
3. **选择集成**：支持 OpenAI、Anthropic、CrewAI、Vercel AI SDK、Pydantic AI 等

#### 更多能力

- **Agent Builder**：无代码可视化构建 Agent
- **提示词工程**：版本管理、协作、快速迭代
- **平台部署**：支持托管云、自托管、混合部署
- **Studio**：可视化端到端设计、测试、优化

#### 合规与安全

支持 HIPAA、SOC 2 Type 2、GDPR 等合规要求，详见 Trust Center。

### 6.4.2 Langfuse

> 原文：[Langfuse 文档](https://langfuse.com/docs)（英文）

Langfuse 是**开源的 LLM 工程平台**，帮助团队协作调试、分析和迭代 LLM 应用。支持自托管，基于 OpenTelemetry，降低厂商锁定风险。

#### 三大核心模块

**1. 可观测性（Observability）**

LLM 应用是非确定性的，传统监控难以覆盖。Langfuse 提供：

- **Trace（追踪）**：记录每次 LLM 调用及检索、嵌入、API 调用等
- **Session（会话）**：追踪多轮对话或 Agent 工作流
- **时间线视图**：排查延迟问题
- **用户级监控**：按 `userId` 追踪成本和用量，可生成深度链接
- **Agent 图**：将 Agent 流程可视化为图

**集成方式**：Python/JS 原生 SDK、50+ 库/框架、OpenTelemetry、或通过 LiteLLM 等网关。

**2. 提示词管理（Prompt Management）**

- 在 UI、API、SDK 中创建、版本、编辑提示词
- 通过标签部署到生产或任意环境，无需改代码
- 在 Playground 中即时测试
- 对比不同版本的延迟、成本、评估指标
- 将提示词与 Trace 关联，了解实际表现

**3. 评估（Evaluation）**

- **实验（Experiments）**：在数据集上系统测试应用
- **数据集**：创建和管理测试集，保证不同场景下的可靠性
- **生产评估**：对生产 Trace 运行评估，及早发现问题
- **评估方式**：LLM-as-a-Judge、用户反馈、人工标注、自定义评估器

**自定义评分示例**：

```python
langfuse.score(
  trace_id="123",
  name="my_custom_evaluator",
  value=0.5,
)
```

#### 为什么选 Langfuse？

- **全平台**：覆盖 LLM 应用开发生命周期
- **多模态**：支持文本、图像等
- **框架支持**：OpenAI SDK、LangChain、LlamaIndex 等
- **SDK 质量**：Python 和 JavaScript 原生 SDK
- **生产优化**：低性能开销
- **开源**：公开 API，可自定义集成

## 总结：串联所有知识点

学完以上资源，你可以建立起这样的认知框架：

```
LLM 应用工程化
├── 架构设计
│   ├── 分层：接入层 → 编排层 → 模型层 → 数据层
│   ├── 挑战：自然语言模糊性、成本、延迟、兼容性
│   ├── 任务编排：Agent、工具、控制流（顺序/并行/条件/循环）
│   └── 工具：LiteLLM 统一多模型调用
├── 对话与体验
│   ├── 流式输出、状态管理、错误处理
│   └── Vercel AI SDK：useChat、streamText、generateText
├── 安全与合规
│   ├── OWASP Top 10：提示词注入、输出处理、供应链等
│   ├── 缓解：输入输出过滤、最小权限、人工审批
│   └── Anthropic：预筛、验证、监控、用户追踪
└── 测试与监控
    ├── 评估：LLM-as-Judge、数据集、实验
    ├── 可观测性：Trace、Session、延迟、成本
    └── 工具：LangSmith、Langfuse
```

**关键认知**：

1. **生产级 ≠ Demo 级**。需要考虑缓存、限流、监控、错误处理、成本控制，以及提示词评估、版本管理、任务编排。

2. **安全是必选项**。提示词注入、不当输出处理、过度代理等风险必须从设计阶段考虑，用多层防护（输入过滤、输出验证、最小权限、人工审批）降低影响。

3. **可观测性是基础**。LLM 输出不确定，需要专门的 Trace、评估、监控工具（LangSmith、Langfuse）来保证质量和可维护性。

4. **统一接口降低复杂度**。LiteLLM、Vercel AI SDK 等工具让你用一套代码支持多模型、多框架，把精力放在业务逻辑上。
