## 2.1 Anthropic 提示词工程指南

> 原文：[Anthropic Prompt Engineering Guide](https://docs.anthropic.com/en/docs/build-with-claude/prompt-engineering/overview)（Anthropic 官方，英文）

Anthropic 是 Claude 模型的开发公司，他们的提示词工程指南是业界公认的系统性教程。核心思想是：**把 Claude 想象成一个"聪明但缺乏背景的新员工"——你交代得越清楚，结果越好**。以下是提炼出的核心要点：

### 基础技巧一：明确、直接地表达

不要指望模型"猜"你想要什么，要**直接说出来**。模糊的指令和精心设计的提示词之间，可能就是"泛泛而谈"和"完全符合需求"的差距。

**对比示例**：

| 模糊写法 | 明确写法 |
|---------|---------|
| "创建一个分析仪表盘" | "创建一个分析仪表盘。包含尽可能多的相关功能和交互，超越基础实现，打造功能完整的版本。" |
| "不要用省略号" | "你的回答会被文字转语音引擎朗读，所以永远不要用省略号，因为语音引擎不知道如何发音。" |

第二个例子特别重要：**解释"为什么"比单纯说"不要做什么"更有效**。模型理解了背后的原因，就能在类似情境下做出更好的判断。

### 基础技巧二：提供上下文和动机

解释你的指令背后的原因，能帮助模型更好地理解目标，从而给出更精准的回答。

**何时需要提供上下文**：
- 说明你在解决什么问题
- 描述输出将如何被使用
- 解释某些限制存在的原因
- 说明输出的目标读者或用途

### 基础技巧三：足够具体

具体性意味着用明确的指导和要求来组织指令。越具体，效果越好。

**具体提示词应包含**：
- 任何要求或限制（饮食需求、预算、技术约束）
- 期望的输出结构（表格、列表、段落）
- 相关背景（受众是谁、目标是什么）
- 清晰的约束（字数、格式、时间线）

### 基础技巧四：善用示例（Few-shot）

示例不是每次都必要，但在解释概念或展示特定格式时非常有用。也叫"少样本学习"——给几个输入输出的例子，让模型"照葫芦画瓢"。

**何时使用示例**：
- 简单指令无法产生稳定结果时
- 任务涉及微妙的模式或惯例
- 需要特定语气或风格
- 期望的格式"展示比描述更容易"

**实用建议**：从一个示例开始（one-shot），只有输出仍不理想时才增加更多（few-shot）。

### 基础技巧五：允许模型表达不确定性

明确告诉模型"如果不确定可以说不知道"，能减少幻觉（一本正经地胡说八道），提高可靠性。

**示例**："分析这份财务数据并找出趋势。如果数据不足以得出结论，请直接说明，不要猜测。"

### 高级技巧一：预填（Prefilling）控制输出格式

预填是指**帮模型开个头**，让它从你指定的内容继续写下去。这在强制输出格式（如 JSON、XML）或跳过开场白时特别有用。

**API 用法示例**：在消息中加一条 assistant 的"假回复"，只包含开头字符：

```
messages=[
    {"role": "user", "content": "从这段产品描述中提取名称和价格，输出为 JSON。"},
    {"role": "assistant", "content": "{"}
]
```

模型会从 `{` 继续写，自然输出纯 JSON，而不会先说"这是您要的 JSON："。

### 高级技巧二：思维链（Chain of Thought）

思维链是指**要求模型在给出答案前一步步推理**。对需要多步分析的复杂任务很有帮助。

**三种常见用法**：

1. **基础版**：直接加"请一步步思考"
2. **引导版**：指定推理阶段，如"先分析……再考虑……最后写出……"
3. **结构化版**：用标签区分推理和最终答案，如 `<thinking>...</thinking>` 和 `<email>...</email>`

### 高级技巧三：提示词链（Prompt Chaining）

把复杂任务拆成多个步骤，每个步骤一个提示词，前一步的输出作为后一步的输入。**用多次调用来换更高的准确率**。

**示例：研究摘要流程**
1. 第一步："总结这篇医学论文的方法论、发现和临床意义"
2. 第二步："审查上面的摘要，评估准确性、清晰度和完整性，给出分级反馈"
3. 第三步："根据以下反馈改进摘要：[第二步的反馈]"

### 常见问题与解决

| 问题 | 解决方案 |
|------|---------|
| 模型只提建议不直接改 | 明确要求行动："修改这个函数"而不是"能建议修改吗？" |
| 模型编造信息 | 明确允许说"我不知道" |
| 输出带多余开场白 | 用预填或明确说"跳过开场，直接给答案" |
| 任务太复杂、结果不稳定 | 拆成多个提示词（链式调用） |
| 输出格式不一致 | 加示例或用预填控制开头 |
| 回答偏题或抓不住重点 | 更明确说明目标，提供背景 |
| 回答太泛泛 | 增加具体性、示例，或明确要求"超越基础" |

### 要避免的误区

- 不要堆砌过时技巧：XML 标签、过度角色设定对现代模型往往不必需
- 不要指望一次写对：第一次提示词很少完美，需要迭代
- 不要一次用上所有技巧：只选能解决当前问题的
- 不要假设模型会读心：要具体，模糊会带来误解
- 不要忽视基础：核心提示词不清楚时，高级技巧也救不了

## 2.2 OpenAI 提示词工程最佳实践

> 原文：[OpenAI Prompt Engineering Best Practices](https://platform.openai.com/docs/guides/prompt-engineering)（OpenAI 官方，英文）

OpenAI 的提示词工程文档面向实际开发，强调**消息角色、可复用提示词、格式化技巧**等工程化实践。以下是核心内容提炼：

### 提示词工程是什么？

提示词工程就是**写出能让模型稳定产出符合要求的指令的过程**。因为模型输出有随机性，写好提示词既是艺术也是科学，但可以借助一些通用技巧提高成功率。

**重要建议**：
- 建立**评测（evals）**来衡量提示词效果，方便迭代或换模型时监控
- 生产环境**固定到具体模型版本**（如 `gpt-4.1-2025-04-14`），保证行为一致

### 消息角色与指令优先级

OpenAI 支持用 `instructions` 参数或消息角色来传递指令，且**不同角色有不同优先级**：

| 角色 | 含义 | 优先级 |
|------|------|--------|
| **developer** | 应用开发者提供的系统规则和业务逻辑 | 最高 |
| **user** | 终端用户提供的输入 | 次之 |
| **assistant** | 模型生成的回复 | - |

可以类比为编程语言：`developer` 像函数定义，`user` 像传入的参数。

### 用 Markdown 和 XML 组织提示词

用 **Markdown 标题、列表** 和 **XML 标签** 划分提示词结构，能帮助模型理解逻辑边界。

**developer 消息的推荐结构**（顺序可调）：
1. **Context（上下文）**：模型需要的额外信息，如私有数据、参考资料
2. **Examples（示例）**：输入输出示例
3. **Instructions（指令）**：规则、该做什么、不该做什么
4. **Identity（身份）**：助手的定位、沟通风格、高层目标

**提示词缓存优化**：把会重复使用的内容放在提示词**开头**，有利于利用提示词缓存，节省成本和延迟。

### Few-shot 学习示例

在 developer 消息中放几个输入输出示例，让模型"照葫芦画瓢"。下面是情感分类的完整示例：

```
# Identity
你是一个帮助标注短评情感的助手，将产品评论标注为 Positive、Negative 或 Neutral。

# Instructions
* 只输出一个词，不要额外格式或说明
* 根据评论情感，只输出 "Positive"、"Negative" 或 "Neutral" 之一

# Examples
<product_review id="example-1">
我太喜欢这副耳机了，音质绝了！
</product_review>
<assistant_response id="example-1">Positive</assistant_response>

<product_review id="example-2">
续航还行，但耳垫感觉廉价。
</product_review>
<assistant_response id="example-2">Neutral</assistant_response>

<product_review id="example-3">
客服太差了，再也不会买。
</product_review>
<assistant_response id="example-3">Negative</assistant_response>
```

### 可复用提示词（Reusable Prompts）

在 OpenAI 控制台可以创建**可复用提示词**，用占位符如 `{{customer_name}}`，在 API 请求中通过 `variables` 传入具体值：

```python
response = client.responses.create(
    model="gpt-5",
    prompt={
        "id": "pmpt_abc123",
        "version": "2",
        "variables": {
            "customer_name": "Jane Doe",
            "product": "40oz juice box"
        }
    }
)
```

这样可以在不改代码的情况下迭代提示词，并支持版本管理。

### 包含相关上下文（RAG）

把额外上下文塞进提示词，让模型基于你指定的资料回答，这种做法常被称为 **RAG（检索增强生成）**。常见做法包括：
- 从向量数据库检索相关文本并放入提示词
- 使用 OpenAI 的**文件搜索工具**基于上传文档生成内容

**上下文窗口**：模型能处理的上下文有上限（以 Token 计），从十几万到百万不等，具体看模型文档。

### GPT 模型 vs 推理模型

| 类型 | 类比 | 提示策略 |
|------|------|---------|
| **GPT 模型** | 像初级同事 | 需要**非常具体**的指令，明确逻辑和数据 |
| **推理模型** | 像资深同事 | 给**高层目标**即可，细节可以交给模型 |

推理模型内部会做"思维链"式的推理，适合复杂任务和多步规划，但通常更慢、更贵。

### 编码任务的最佳实践

- **明确角色和工作流**：把模型设定为软件工程智能体，说明工具使用方式
- **测试与验证**：要求用单元测试或命令验证修改，注意 `apply_patch` 等工具可能失败也返回 "Done"
- **工具使用示例**：给出具体的函数调用示例，提高可靠性
- **Markdown 规范**：要求用代码块、列表、表格等规范输出

## 2.3 Prompt Engineering Guide（提示工程指南）

> 原文：[Prompt Engineering Guide](https://www.promptingguide.ai/zh)（DAIR.AI 社区维护，有中文版）

这是一份社区维护的**提示词工程百科全书**，涵盖零样本、少样本、思维链等核心概念，并配有大量示例。以下是核心内容提炼：

### 提示工程是什么？

提示工程是一门较新的学科，关注**提示词的开发和优化**，帮助用户把大语言模型用在各种场景。它不仅是写提示词，还包含与 LLM 交互、理解其能力与局限的整套技能。

**两大用途**：
- **研究**：提升 LLM 在问答、算术推理等复杂任务上的能力
- **开发**：设计工程技术，与 LLM 或相关工具高效对接

### 少样本提示（Few-shot Prompting）

虽然 LLM 有不错的零样本能力，但在更复杂任务上，**在提示词里放几个示例**往往能明显提升效果。这种"给几个例子让模型模仿"的方式叫少样本提示。

**经典示例**：教模型用新词造句

```
"whatpu"是坦桑尼亚的一种小型毛茸茸的动物。一个使用 whatpu 这个词的句子的例子是：
我们在非洲旅行时看到了这些非常可爱的 whatpus。

"farduddle"是指快速跳上跳下。一个使用 farduddle 这个词的句子的例子是：
```

模型通过一个示例就能学会任务模式。任务越难，可以适当增加示例数量（3-shot、5-shot 等）。

**少样本的注意事项**（来自 Min 等人研究）：
- 示例的格式对效果影响很大，即使标签随机，有格式也比没有好
- 标签空间和输入文本的分布都很重要
- 从真实标签分布中选示例比均匀分布更有帮助

### 少样本的局限：需要思维链

对于**多步推理**任务，单纯少样本可能不够。例如：

**问题**：这组数字中的奇数加起来是偶数吗？15、32、5、13、82、7、1

即使用 4 个示例，模型仍可能直接答"True"或"False"而不展示推理过程，且容易答错。**思维链（CoT）** 正是为解决这类问题而提出的——让模型先写出推理步骤，再给答案。

### 思维链提示（Chain-of-Thought）

思维链通过在提示中**加入中间推理步骤**，显著提升复杂推理能力。可与少样本结合使用。

**示例**：奇数求和是否为偶数

```
这组数中的奇数加起来是偶数：4、8、9、15、12、2、1。
A：将所有奇数相加（9、15、1）得到 25。答案为 False。

这组数中的奇数加起来是偶数：17、10、19、4、8、12、24。
A：将所有奇数相加（17、19）得到 36。答案为 True。

...

这组数中的奇数加起来是偶数：15、32、5、13、82、7、1。
A：
```

模型输出：`将所有奇数相加（15、5、13、7、1）得到 41。答案为 False。`

有时**一个带推理步骤的示例**就足够，说明大模型具备"从示例中学习推理模式"的能力。

### 零样本思维链（Zero-shot CoT）

Kojima 等人提出：在提示末尾加一句**"让我们逐步思考"**（或英文 "Let's think step by step"），即使不给示例，也能激发模型的逐步推理。

**示例**：

```
我去市场买了 10 个苹果。我给了邻居 2 个和修理工 2 个。然后我又买了 5 个并吃了 1 个。我还剩下多少苹果？
```

不加这句时，模型可能直接答"11 个"（错误）。加上"让我们逐步思考"后：

```
首先，您从 10 个苹果开始。
您给了邻居和修理工各 2 个，所以还剩下 6 个。
然后您买了 5 个，现在有 11 个。
最后您吃了 1 个，所以还剩下 10 个。
```

答案正确，且推理过程清晰。这在**没有合适示例**时特别有用。

### 自动思维链（Auto-CoT）

手工设计思维链示例比较费劲。Auto-CoT 用模型自动生成示例，主要两步：
1. **问题聚类**：把问题分成若干组
2. **演示抽样**：从每组选代表性问题，用"让我们一步一步地思考"生成推理链作为示例

通过**问题多样性**和**简单启发式**（如限制问题长度、推理步数）来减轻自动生成中的错误影响。

## 2.4 Chain-of-Thought Prompting 原始论文

> 原文：[Chain-of-Thought Prompting Elicits Reasoning in Large Language Models](https://arxiv.org/abs/2201.11903)（Wei 等人，Google，2022）

这篇论文首次系统性地提出**思维链（Chain-of-Thought, CoT）** 提示方法，是理解 CoT 的学术源头。

### 核心结论

**生成一系列中间推理步骤（思维链），能显著提升大语言模型在复杂推理任务上的表现。**

具体做法很简单：在提示中提供**几个带推理过程的示例**（few-shot CoT），模型就会在回答时也展示推理步骤，从而在算术、常识、符号推理等任务上取得更好效果。

### 实验数据

论文在三个大模型上做了实验，结论包括：

- **GSM8K 数学应用题**：仅用 8 个思维链示例提示 540B 参数模型，就能达到当时最优水平，甚至超过专门微调过的 GPT-3+验证器
- 收益非常明显：在某些任务上，从"直接给答案"切换到"先写推理再给答案"，准确率有大幅提升

### 为什么有效？

1. **分解复杂问题**：多步推理被拆成一步步，每步更简单，出错概率降低
2. **利用预训练中的推理模式**：大模型在训练数据里见过大量"因为……所以……"式的推理，CoT 激活了这类模式
3. **涌现能力**：作者指出，这种推理能力在**足够大的模型**上才会明显出现，是小模型不具备的"涌现"特性

### 与少样本的关系

- **普通少样本**：只给"输入→输出"示例，模型容易在复杂推理上直接猜答案
- **思维链少样本**：示例中包含**推理过程**，模型学会先推理再作答

论文表明，对于需要多步推理的任务，**示例中是否包含推理步骤**，对效果影响巨大。

### 实践启示

1. 遇到数学、逻辑、多步分析类任务，优先尝试 CoT
2. 可以从零样本 CoT 开始（加"请一步步思考"），不够再补 few-shot 示例
3. 示例中的推理要清晰、正确，避免把错误模式教给模型

## 2.5 LangChain Prompt Templates 文档

> 原文：[LangChain Prompt Templates 文档](https://python.langchain.com/docs/concepts/prompt_templates/)（LangChain 官方，英文）

LangChain 是流行的 LLM 应用开发框架，其**提示词模板（Prompt Templates）** 是工程化管理提示词的标准方式。核心思路：**把提示词写成带占位符的模板，运行时再填入具体变量**。

### 为什么需要提示词模板？

在实际开发中：
- 同一类任务（如代码审查、文档生成）会反复出现
- 每次只改少量内容（如文件名、代码片段），主体结构不变
- 需要团队协作、版本管理、复用

模板化可以避免重复写大段提示词，保证输出风格一致，也便于维护。

### PromptTemplate 核心概念

**PromptTemplate** 是基础模板类，主要特性包括：

| 特性 | 说明 |
|------|------|
| **input_variables** | 模板需要的变量名列表，如 `["topic", "code"]` |
| **template** | 带占位符的字符串，如 `"请审查以下 {language} 代码：\n\n{code}"` |
| **format()** | 传入变量值，返回填充后的完整提示词 |
| **partial()** | 预先填充部分变量，返回一个新模板，其余变量稍后填充 |
| **save()** | 将模板保存到文件（如 YAML），便于版本管理 |

### 基本用法示例

```python
from langchain_core.prompts import PromptTemplate

# 从字符串创建模板，自动推断变量
prompt = PromptTemplate.from_template(
    "你是一位 {role}。请分析以下内容：\n\n{content}"
)

# 填充变量
filled_prompt = prompt.format(role="资深工程师", content="某段代码...")
print(filled_prompt)
```

### 部分填充（Partial）

当有些变量在创建时就已知（如当前日期），可以用 `partial` 预填：

```python
from datetime import datetime

def get_datetime():
    return datetime.now().strftime("%Y-%m-%d %H:%M")

prompt = PromptTemplate(
    template="今天是 {date}。你是一位 {role}，请分析：{content}",
    input_variables=["role", "content"],
    partial_variables={"date": get_datetime}
)

# 使用时只需传入 role 和 content
filled = prompt.format(role="产品经理", content="用户反馈...")
```

### ChatPromptTemplate：对话式模板

对于对话模型，需要区分 system、user、assistant 等角色。**ChatPromptTemplate** 专门处理这种结构：

```python
from langchain_core.prompts import ChatPromptTemplate, SystemMessagePromptTemplate, HumanMessagePromptTemplate

template = ChatPromptTemplate.from_messages([
    ("system", "你是{role}，请用{style}风格回答。"),
    ("human", "{question}")
])

messages = template.format_messages(role="技术顾问", style="简洁", question="什么是 RAG？")
# 得到 [SystemMessage(...), HumanMessage(...)]，可直接传给 Chat 模型
```

### 常用模板格式

LangChain 支持多种占位符语法：
- **f-string**：`{variable}`（默认）
- **Jinja2**：`{{ variable }}`
- **Mustache**：`{{variable}}`

可根据项目习惯选择。

### 提示词链（Prompt Chain）

多个模板可以串联：前一个模板的输出作为后一个的输入。例如：

1. 模板 A：根据主题生成大纲
2. 模板 B：根据大纲生成完整文章

这对应前文提到的"提示词链"概念，适合多步骤任务。

### 工程化建议

- **模板与代码分离**：把模板放在 YAML/JSON 或独立文件中，便于非开发人员修改
- **版本管理**：对模板做 Git 管理，记录每次修改和效果变化
- **与 LangSmith 结合**：用 LangSmith 追踪不同模板的调用效果，做 A/B 测试

## 总结：串联所有知识点

学完以上 5 个资源，可以形成这样的知识框架：

```
提示词工程知识体系
├── 基础原则（Anthropic + OpenAI）
│   ├── 明确、具体、直接
│   ├── 提供上下文和"为什么"
│   ├── 善用示例（Few-shot）
│   ├── 允许模型说"不知道"
│   └── 用 Markdown/XML 组织结构
├── 高级技巧（Prompting Guide + CoT 论文）
│   ├── 少样本学习：给例子让模型模仿
│   ├── 思维链：让模型一步步推理
│   ├── 零样本 CoT："让我们逐步思考"
│   └── 提示词链：多步任务拆成多个提示词
└── 工程化（LangChain）
    ├── 模板变量与复用
    ├── 部分填充
    ├── 对话模板（ChatPromptTemplate）
    └── 版本管理与追踪
```

**关键认知**：

1. **好的提示词 = 好的沟通**。把模型当成聪明但缺背景的同事，说得越清楚，结果越好。模糊指令和精准指令的差距，往往就是"泛泛而谈"和"完全符合需求"的差距。

2. **示例的力量**。对于格式、风格、复杂规则，展示几个例子往往比大段描述更有效。从 one-shot 开始，不够再加。

3. **思维链是推理任务的利器**。数学、逻辑、多步分析类问题，加上"请一步步思考"或给带推理的示例，效果常有明显提升。

4. **工程化从模板开始**。用占位符、变量、模板文件管理提示词，能提高一致性、可维护性和团队协作效率。

5. **迭代比一次写对更重要**。第一次提示词很少完美，要测试、调整、再测试。建立简单评测，固定模型版本，才能稳定迭代。
