## 二、技术选型方案

### 2.1 气象 API 接入（自动获取气温/气压/风向）

| 方案 | 免费额度 | 核心参数 | 推荐理由 |
|------|---------|---------|---------|
| **和风天气 API**（首选） | 每日 1000 次 | 气温、体感温度、气压、湿度、风向、风速、降水量、日出日落、紫外线 | 数据全面，含气压（钓鱼关键参数），文档完善，支持逐小时预报 |
| **易客云天气 API**（备选） | 每日 10 万次 | 气温、气压、湿度、风向风速、空气质量 | 免费额度大，适合初期快速验证 |
| **中国气象局 API**（兜底） | 无限制 | 气温、气压、风力、降水量 | 官方数据源，稳定可靠，无调用限制 |

核心需求：支持中文、支持 JSON 结构化输出、免费或极低成本、无需自建服务器、响应速度够快（APP场景）。

#### 各方案详细说明

**1. DeepSeek V3.2（首选主力）**

| 项目 | 说明 |
|------|------|
| 接入方式 | `https://api.deepseek.com/v1/chat/completions`，兼容 OpenAI SDK |

| JSON 模式 | 原生支持 `response_format: { type: "json_object" }`，无需额外 prompt 引导 |
| 中文能力 | 中文训练数据占比高，钓鱼领域中文理解准确 |
| 上下文窗口 | 128K token，足够携带完整用户画像+气象数据+历史记录 |
| 缓存命中折扣 | 重复 prompt 前缀可享 90% 输入折扣（System Prompt 固定部分几乎免费） |

#### 成本估算（月活 1000 用户）

```
假设：
- 每用户日均查询 2 次（综合推荐）
- 每次查询消耗约 3000 token（输入1500 + 输出1500）
- 每用户月均反馈 4 次（消耗约 1000 token/次）

月总消耗：
  综合推荐：1000人 × 2次/天 × 30天 × 3000 token = 1.8 亿 token
  反馈解析：1000人 × 4次/月 × 1000 token = 400 万 token

免费额度覆盖：
  DeepSeek 免费 100 万 token/月 → 覆盖约 333 次综合推荐
  阿里百炼 免费 200 万 token   → 覆盖约 666 次
  硅基流动 永久免费            → 无限次（7B模型）

结论：
  ┌─────────────────────────────────────────────────────┐
  │ 初期（<100 用户）：三家免费额度完全覆盖，零成本运行     │
  │ 中期（100-500用户）：核心功能用付费模型≈$15-30/月      │
  │ 后期（1000+用户）：混合调度，≈$50-80/月               │
  │                                                     │
  │ 对比：同等规模用 GPT-4o 约需 $300-500/月              │
  └─────────────────────────────────────────────────────┘
```

#### 为什么不选其他模型

| 模型 | 不选原因 |
|------|---------|
| GPT-4o / GPT-5 | 价格是 DeepSeek 的 5-20 倍，中文能力不如国产模型，需翻墙 |
| Claude Sonnet | 价格高，无免费层，需翻墙 |
| Google Gemini | 免费层慷慨但国内需翻墙，APP用户无法直连，延迟不可控 |
| Groq（Llama系列） | 速度极快但中文能力弱，钓鱼领域中文术语理解差，且需翻墙 |
| 本地部署（Ollama等） | 需要 GPU 服务器，7B 模型质量不够，成本远高于 API 调用 |

## 三、用户画像打标系统

### 3.1 打标原则：自动为主，手动为辅

```
核心理念：用户不需要填问卷，系统通过观察行为自动完成 90% 的画像构建。
用户只在"系统无法推断"或"想主动纠偏"时才需要手动操作。

打标数据来源优先级：
  ① 行为自动推断（最优先，覆盖最广）
  ② 反馈数据沉淀（用户提交渔获后自动提取）
  ③ 用户主动设置（设置页可选，非强制）
  ④ 默认值兜底  （冷启动时使用，随数据积累逐步替换）
```

### 3.4 打标评分机制

#### 标签成熟度评分（替代原"可信度分数"）

每个标签独立计算成熟度，而非整个画像一个分数。这样更精细：某些标签数据充足可以强个性化，另一些数据不足则回退通用推荐。

```
单标签成熟度 = min(1.0, 有效数据点数 / 该标签所需最小数据点) × 数据新鲜度系数

其中：
- 有效数据点数：该标签累计收到的有效信号次数
- 最小数据点：见上方各标签的 min_data_points（默认5）
- 数据新鲜度系数：
  - 最近7天内有更新  = 1.0
  - 7-30天内有更新   = 0.8
  - 30-90天内有更新  = 0.5
  - 超过90天未更新   = 0.3（标记为"陈旧"，提示用户确认）
```

#### 整体画像置信度（用于推荐引擎）

```
画像置信度 = Σ(单标签成熟度 × 标签权重) / Σ(标签权重)

加权汇总所有标签的成熟度，得出一个 0-1 的综合分数。
```

#### 数据来源可信度加权

```
同一标签有多个数据来源时，按可信度加权融合：

来源类型          可信度系数    说明
──────────────────────────────────────────────────
渔获反馈数据       1.0         用户实际钓鱼后的真实数据，最可靠
用户主动设置       0.9         用户明确表达的偏好，但可能有认知偏差
导航/地图行为      0.8         实际行动信号，接近真实
查询行为           0.6         有意图但不一定付诸行动
浏览行为           0.4         最弱信号，可能只是好奇
默认值             0.1         无数据时的兜底，尽快替换
```

```
    │   └→ 气温、气压(含趋势)、风向风速、日出日落、降水概率
    │
    ├─③ 用户画像系统：读取用户标签 + 计算置信度分数
    │   └→ 经验等级、钓法偏好、鱼种偏好、时段偏好、中鱼率
    │
    ├─④ 用户输入：选择水域类型 + 可选目标鱼种
    │
    ▼
  参数组装 → 填充 Prompt 模板变量
    │
    ├→ 调用模块B（综合推荐）→ 获取基础推荐
    ├→ 调用模块D（个性化引擎）→ 重排序
    ├→ 调用模块C（水层数据）→ 渲染剖面图
    │
    ▼
  结果展示（卡片+时间轴+水层图+钓鱼指数）
    │
    ▼
  用户出钓 → 反馈渔获
    │
    ├→ 调用模块E（反馈解析）→ 结构化数据
    ├→ 更新用户画像标签
    ├→ 触发合理性校验规则
    └→ 画像置信度重新计算
```

## 六、关键设计决策备注

| 决策点 | 选择 | 理由 |
|--------|------|------|
| 气压数据 | 必须接入，不可省略 | 气压是影响鱼口的第一要素，比气温更重要 |
| 画像冷启动 | 注册问卷(3题) + 通用推荐兜底 | 避免新用户体验差，3题快速建立基础画像 |
| 置信度透明 | 向用户展示"为什么这样推荐" | 建立信任，用户理解推荐逻辑后更愿意反馈 |
| 异常检测 | 软警告为主，不直接拒绝 | 钓鱼是个人爱好，异常标签可能是用户真实但小众的偏好 |
| LLM vs 规则引擎 | 混合架构 | 鱼种-水层基础数据用本地规则引擎(快+稳)，个性化推荐和自然语言交互用LLM |
| 离线模式 | 本地缓存核心鱼种库+最近一次气象数据 | 钓鱼场景常在信号差的野外 |
