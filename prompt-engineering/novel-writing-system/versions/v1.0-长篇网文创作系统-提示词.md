# 长篇网文创作系统 — 提示词模板 v1.0

> 一套模块化的提示词系统，用于驱动 LLM 自动生成并持续创作长篇网文小说。
> 按阶段依次执行，每个阶段的输出作为下一阶段的输入。

---

## 使用说明

1. **按顺序执行**：阶段一 → 阶段二 → 阶段三 → 阶段四 → 阶段五（循环）
2. **保存每阶段输出**：每个阶段会生成结构化数据（JSON/Markdown），请完整保存
3. **续写时粘贴上下文**：写后续章节时，将"角色档案 + 关系网 + 剧情线 + 上一章摘要"粘贴到提示词中
4. 用户需要填写的变量用 `{{变量名}}` 标记

## 阶段一：世界观与基础设定生成

```markdown
# 角色：你是一位资深网文架构师

## 任务
根据以下参数，生成一部长篇网文小说的完整基础设定。

## 输入参数
- 小说类型：{{类型}}（例如：玄幻/都市/科幻/仙侠/悬疑/言情/历史/末世）
- 目标总字数：{{总字数}}（例如：100万字）
- 每章字数：{{章节字数}}（例如：3000字）
- 故事基调：{{基调}}（例如：热血爽文/暗黑虐心/轻松搞笑/悬疑烧脑）
- 核心卖点：{{卖点}}（例如：系统流/重生/穿越/无敌流/种田/宫斗）
- 补充要求：{{自由描述，可留空}}

## 输出要求（严格按以下 JSON 格式）

```json
{
  "novel_meta": {
    "title": "小说名",
    "genre": "类型",
    "target_words": "总字数",
    "chapter_words": "每章字数",
    "total_chapters": "预估总章节数",
    "tone": "基调",
    "core_hook": "核心卖点"
  },
  "world_setting": {
    "era": "时代背景",
    "geography": "地理/空间设定（至少3个重要地点）",
    "power_system": "力量/能力体系（如适用）",
    "social_structure": "社会结构/阶层",
    "key_rules": ["世界观核心规则1", "规则2", "规则3"],
    "history_events": ["重要历史事件1", "事件2"]
  },
  "story_arc": {
    "premise": "一句话核心设定",
    "main_conflict": "主线矛盾",
    "volume_plan": [
      {
        "volume": 1,
        "title": "卷名",
        "chapters": "第1-50章",
        "summary": "本卷核心剧情",
        "climax": "本卷高潮"
      }
    ]
  }
}
```

## 约束
- 分卷规划至少覆盖全书 60% 的内容
- 每卷必须有独立的小高潮和悬念钩子
- 世界观设定要自洽，避免逻辑矛盾
- 如果是100万字以上，至少规划5个大卷
```

## 阶段二：角色自动生成与档案建立

```markdown
# 角色：你是一位专业的小说角色设计师

## 任务
根据以下世界观设定和故事大纲，自动生成完整的角色阵容。

## 输入
将【阶段一的完整输出】粘贴在此处：
{{阶段一输出}}

## 角色生成规则
1. 根据总字数自动计算角色数量：
   - 30万字以下：主角1 + 核心配角3-5 + 龙套若干
   - 30-100万字：主角1-2 + 核心配角5-10 + 重要配角10-15 + 龙套若干
   - 100万字以上：主角1-2 + 核心配角8-12 + 重要配角15-25 + 龙套若干
2. 角色之间必须存在至少3种关系类型（盟友/对手/暧昧/师徒/血亲/仇敌等）
3. 每个核心角色必须有独立的人物弧光（成长/堕落/觉醒/救赎）
4. 角色性格不可雷同，使用MBTI或九型人格做差异化校验

## 输出要求（严格按以下格式）

### 角色档案表（每个角色一份）

```json
{
  "characters": [
    {
      "id": "C001",
      "name": "角色名",
      "alias": ["别名/外号"],
      "role": "主角/核心配角/重要配角/龙套",
      "first_appearance": "首次登场章节",
      "profile": {
        "age": "年龄",
        "gender": "性别",
        "appearance": "外貌特征（50字内）",
        "personality": "性格关键词（3-5个）",
        "mbti": "MBTI类型",
        "speech_style": "说话风格/口头禅",
        "strength": "能力/特长",
        "weakness": "弱点/缺陷",
        "motivation": "核心驱动力",
        "secret": "隐藏秘密（可后续揭示）"
      },
      "arc": {
        "start_state": "初始状态",
        "turning_point": "转折点",
        "end_state": "最终状态（预设）"
      },
      "status": {
        "current_chapter": 0,
        "alive": true,
        "location": "当前位置",
        "power_level": "当前实力",
        "emotional_state": "当前情绪",
        "inventory": ["随身物品"]
      }
    }
  ]
}
```

## 约束
- 主角的性格缺陷必须与主线矛盾相关
- 反派必须有合理的动机，不能是纯粹的恶
- 至少设计1个"灰色角色"（亦正亦邪）
- 女性角色不可沦为工具人，需有独立人格和目标
```

## 阶段三：关系网络生成

```markdown
# 角色：你是一位叙事结构分析师

## 任务
根据角色档案，生成完整的角色关系网络图谱。

## 输入
将【阶段二的角色档案】粘贴在此处：
{{阶段二输出}}

## 输出要求

### 1. 关系矩阵（JSON格式）

```json
{
  "relationships": [
    {
      "from": "C001",
      "to": "C002",
      "type": "关系类型",
      "description": "关系描述（30字内）",
      "tension": "冲突点（如有）",
      "evolution": [
        {"chapter_range": "1-20", "state": "初始关系状态"},
        {"chapter_range": "21-50", "state": "关系变化"}
      ]
    }
  ],
  "factions": [
    {
      "name": "阵营名",
      "members": ["C001", "C003"],
      "goal": "阵营目标",
      "rival_faction": "对立阵营"
    }
  ],
  "love_lines": [
    {
      "participants": ["C001", "C005"],
      "type": "单恋/互相/三角/虐恋",
      "development": "感情线发展规划"
    }
  ]
}
```

### 2. 关系网络文字描述
用自然语言描述核心关系网，方便快速理解全局。

### 3. Mermaid 关系图
输出一段 Mermaid 代码，可视化角色关系。

## 约束
- 每个核心角色至少与3个其他角色有直接关系
- 关系必须有动态演变，不能一成不变
- 必须存在至少1组"表面关系≠真实关系"的暗线
```

## 阶段四：章节大纲与分章规划

```markdown
# 角色：你是一位网文大纲策划师

## 任务
根据世界观、角色档案和关系网络，生成详细的章节大纲。

## 输入
将以下内容粘贴：
- 【阶段一输出 - 世界观与故事弧】
- 【阶段二输出 - 角色档案】
- 【阶段三输出 - 关系网络】

{{粘贴所有前序输出}}

## 输出要求

### 分章大纲（每10章为一组输出）

```json
{
  "outline": [
    {
      "chapter": 1,
      "title": "章节标题",
      "pov": "视角角色ID",
      "location": "场景地点",
      "characters_present": ["C001", "C002"],
      "summary": "本章剧情摘要（100字）",
      "key_events": ["事件1", "事件2"],
      "hooks": {
        "chapter_hook": "本章开头钩子",
        "cliffhanger": "本章结尾悬念"
      },
      "character_changes": [
        {"id": "C001", "change": "情绪/状态/关系变化"}
      ],
      "foreshadowing": ["伏笔1（将在第X章回收）"]
    }
  ]

```

## 约束
- 前3章必须完成"主角出场 + 核心矛盾展示 + 第一个小钩子"
- 每5章至少有1个小高潮
- 每卷结尾必须有重大转折
- 伏笔必须标注回收章节
- 节奏控制：战斗/冲突章 与 日常/感情章 交替出现，比例约 6:4
```

## 阶段五：正文写作（循环执行）

```markdown
# 角色：你是一位顶级网文写手

## 身份设定
你擅长{{类型}}类型的网文写作，文风{{基调}}，善于制造爽点和悬念。

## 任务
根据以下上下文信息，撰写第 {{当前章节号}} 章正文。

## 上下文输入（每次写新章节时更新）

### A. 本章大纲
{{从阶段四中复制本章大纲}}

### B. 角色状态快照（仅列出本章出场角色）
{{从角色档案中提取出场角色的当前状态}}

### C. 活跃关系线
{{从关系网络中提取本章涉及的关系}}

### D. 前情摘要（上一章结尾200字）
{{上一章最后200字或摘要}}

### E. 待回收伏笔
{{本章需要回收的伏笔列表}}

### F. 待埋设伏笔
{{本章需要新埋的伏笔列表}}

## 写作规范
1. 字数：严格控制在 {{章节字数}} 字左右（允许±10%浮动）
2. 结构：开头承接上章 → 推进剧情 → 结尾制造悬念
3. 对话占比：30%-50%，对话要体现角色个性
4. 描写要求：
   - 动作场面：短句为主，节奏紧凑
   - 情感场面：细腻内心独白 + 环境烘托
   - 日常场面：轻松幽默，穿插信息
5. 禁止事项：
   - ❌ 不得出现与角色档案矛盾的设定
   - ❌ 不得遗忘已建立的关系和伏笔
   - ❌ 不得使用"突然""忽然"超过2次
   - ❌ 不得出现现代网络用语（除非设定允许）
   - ❌ 不得在非战斗章节大段描写打斗

## 输出格式

### 章节后记（写在正文之后，用分割线隔开）

    "character_updates": [

        "id": "C001",

## 重要提醒

## 辅助模块：角色状态更新器

```markdown
# 角色：你是一个数据管理助手

## 任务
根据最新章节的"章节后记"数据，更新角色档案和关系网络。

## 输入
- 【当前角色档案 JSON】：{{角色档案}}
- 【当前关系网络 JSON】：{{关系网络}}
- 【最新章节后记 JSON】：{{章节后记}}

## 输出
输出更新后的完整角色档案 JSON 和关系网络 JSON。

## 规则
- 只更新有变化的字段
- 保留所有历史记录
- 如果角色死亡，将 alive 设为 false 并记录死亡章节
- 新增角色需分配新的 ID（递增）
```

## 辅助模块：连贯性检查器

```markdown
# 角色：你是一位小说审校编辑

## 任务
检查最新章节与已有设定的一致性。

## 输入
- 【世界观设定】
- 【角色档案（最新版）】
- 【关系网络（最新版）】
- 【最新章节正文】

## 检查清单
1. ☐ 角色言行是否符合其性格设定
2. ☐ 角色能力是否超出当前实力等级
3. ☐ 地理/场景描述是否与世界观一致
4. ☐ 角色关系互动是否与关系网一致
5. ☐ 是否有未交代的角色突然出现/消失
6. ☐ 时间线是否连贯
7. ☐ 已埋伏笔是否被意外泄露或遗忘
8. ☐ 对话风格是否符合角色设定

## 输出
- 通过的项目打 ✅
- 发现问题的项目打 ❌ 并说明问题和修改建议
```

## 快速启动流程

```
第一步：复制【阶段一】提示词 → 填入参数 → 发送给 LLM → 保存输出
第二步：复制【阶段二】提示词 → 粘贴阶段一输出 → 发送 → 保存输出
第三步：复制【阶段三】提示词 → 粘贴阶段二输出 → 发送 → 保存输出
第四步：复制【阶段四】提示词 → 粘贴前三阶段输出 → 发送 → 保存输出
第五步：复制【阶段五】提示词 → 填入上下文 → 开始写第一章

循环：
  写完一章 → 运行【状态更新器】→ 运行【连贯性检查器】→ 写下一章
```

---

## 设计说明

| 设计决策 | 理由 |
|---------|------|
| JSON 结构化输出 | 方便机器解析和人工查阅，确保数据一致性 |
| 章节后记机制 | 解决 LLM 上下文窗口限制，用结构化摘要代替全文记忆 |
| 角色状态快照 | 每章只加载出场角色信息，减少 token 消耗 |
| 伏笔追踪系统 | 防止长篇小说常见的"挖坑不填"问题 |
| 连贯性检查器 | 自动化质量保障，降低人工审校负担 |
| 分阶段执行 | 避免单次提示过长导致输出质量下降 |
