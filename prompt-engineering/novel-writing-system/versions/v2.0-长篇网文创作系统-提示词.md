# 长篇网文创作系统 — 提示词模板 v2.0

> 一套模块化的提示词系统，用于驱动 LLM 自动生成并持续创作长篇网文小说。
> 按阶段依次执行，每个阶段的输出作为下一阶段的输入。
> v2.0 新增：类型差异化写作引擎、爽感节奏曲线、系统流创新校验、规则类逻辑框架。

---

## 使用说明

1. **按顺序执行**：阶段一 → 阶段二 → 阶段三 → 阶段四 → 阶段五（循环）
2. **保存每阶段输出**：每个阶段会生成结构化数据（JSON/Markdown），请完整保存
3. **续写时粘贴上下文**：写后续章节时，将"角色档案 + 关系网 + 剧情线 + 上一章摘要"粘贴到提示词中
4. 用户需要填写的变量用 `{{变量名}}` 标记

---

## 前置模块：类型差异化写作引擎（阶段一之前必读）

> 本模块定义了不同小说类型的专属写作策略。阶段一填入类型后，系统会自动激活对应策略，
> 贯穿后续所有阶段。

### 引擎 A：鬼怪规则类（恐怖/灵异/无限流/规则怪谈）

**核心原则：逻辑即恐惧。规则越清晰，违反规则的后果越恐怖。**

```yaml
writing_engine: "rule_horror"
priorities:
  - 逻辑自洽 > 氛围渲染 > 想象力
  - 规则必须可验证：读者能根据已知规则推导出"谁会死"
  - 恐惧来源于"已知的确定性"而非"未知的模糊感"

rule_system_design:
  structure: "三层规则体系"
  layers:
    - name: "表层规则"
      desc: "角色和读者最先发现的规则，通常不完整或有误导性"
      example: "晚上12点不能照镜子"
      reveal_timing: "前3章内展示"
    - name: "深层规则"
      desc: "隐藏在表层之下的真实运作逻辑"
      example: "镜子不是禁忌本身，而是'被注视'才会触发诅咒"
      reveal_timing: "中期反转时揭示"
    - name: "元规则"
      desc: "整个怪谈世界的底层法则，解释为什么规则存在"
      example: "所有规则都是某个意识体的'游戏条款'"
      reveal_timing: "大结局或关键转折"

  logic_constraints:
    - 每条规则必须有明确的触发条件、生效范围、违反后果
    - 规则之间不能矛盾，但可以存在"看似矛盾实则不同层级"的情况
    - 主角破解规则的方式必须基于已给出的线索，禁止凭空开挂
    - 每次有角色死亡/遭遇危险，读者回溯时必须能找到"他违反了哪条规则"

  tension_formula:
    - "信息差制造悬念：读者知道规则但角色不知道 → 紧张感"
    - "规则冲突制造绝境：遵守A规则就必然违反B规则 → 两难"
    - "规则迭代制造升级：旧规则被破解，新规则更致命 → 层层加码"

  chapter_rhythm:
    - "规则发现章 → 规则试探章 → 规则验证章（有人因此遇难）→ 规则破解章 → 新规则出现"
    - 每5-8章完成一个"规则循环"
    - 禁止连续3章以上没有规则相关推进

  forbidden:
    - ❌ 规则模糊不清，无法被读者理解和推导
    - ❌ 主角靠"灵光一闪"解决问题，没有逻辑铺垫
    - ❌ 鬼怪的行为前后不一致（除非有更高层规则解释）
    - ❌ 恐怖场景只靠jump scare式描写，没有规则支撑
    - ❌ 角色明知规则还犯蠢（除非有合理的心理/情境压力解释）
```

### 引擎 B：玄幻 / 科幻类（仙侠/奇幻/星际/赛博朋克）

**核心原则：想象力是第一生产力。设定必须"从未见过"但"细想合理"。**

```yaml
writing_engine: "imagination_first"
priorities:
  - 新颖性 > 合理性 > 逻辑严密性
  - 设定要让读者产生"卧槽还能这样"的惊叹感
  - 合理性是底线，不是天花板——在合理范围内尽量突破想象边界

worldbuilding_innovation:
  method: "逆向设定法"
  steps:
    - "列出该类型最常见的10个设定套路"
    - "逐一反转或融合，生成候选设定"
    - "用'三秒钟测试'筛选：读者看到设定的前三秒是否会产生好奇心"
    - "用'深挖测试'验证：这个设定能否支撑至少3个独立的剧情冲突"

  innovation_checklist:
    - ☐ 力量体系是否有独特的获取/升级方式（不是简单的打怪升级）
    - ☐ 世界观是否有至少1个"反直觉但自洽"的核心设定
    - ☐ 种族/阵营划分是否超越了"人类vs异族"的二元对立
    - ☐ 科技/法术的代价体系是否有创意（不是简单的消耗灵力/能量）
    - ☐ 地理/空间设定是否有独特的物理法则或生态逻辑

  anti_cliche_rules:
    - "如果一个设定在起点/晋江排名前100的同类小说中出现过3次以上，必须变体"
    - "修炼体系禁止使用：炼气→筑基→金丹→元婴 这类标准模板，除非有本质性改造"
    - "科幻设定禁止使用：人类联邦vs外星帝国 这类标准框架，除非有独特的政治/文化维度"

  imagination_amplifiers:
    - "跨领域嫁接：将现实中某个冷门领域的逻辑移植到奇幻世界"
    - "尺度跳跃：在微观/宏观之间切换视角，制造认知冲击"
    - "规则颠覆：让读者以为理解了世界规则，然后揭示更大的框架"

  forbidden:
    - ❌ 力量体系只是换了名字的"打怪升级"
    - ❌ 世界观是"中世纪欧洲+魔法"或"古代中国+修仙"的简单叠加
    - ❌ 外星文明只是"长得不同的人类"
    - ❌ 科技设定没有社会影响（技术改变世界观，不只是武器）
```

### 引擎 C：系统流 / 穿越类

**核心原则：系统是工具不是万能钥匙。系统的独特性决定小说的独特性。**

```yaml
writing_engine: "system_innovation"
priorities:
  - 系统创新性 > 爽感 > 合理性
  - 系统必须有"从未在网文中出现过"的核心机制
  - 系统强大但有明确边界，不能解决所有问题

system_design_protocol:
  step1_innovation_generation:
    method: "领域迁移法"
    process:
      - "从现实世界选择一个非游戏领域（如：拍卖、烹饪、社交信用、天气预报、股票交易、考古、快递物流...）"
      - "提取该领域的核心机制（如拍卖：竞价、底价、流拍、暗拍）"
      - "将机制转化为系统功能（如：主角的能力需要'竞拍'获得，出价是寿命/记忆/情感）"
      - "设计系统的独特交互方式（不是简单的面板+任务+奖励）"

  step2_balance_check:
    power_ceiling: "系统的最强能力不能超过当前世界观天花板的70%"
    growth_curve: "前期系统提供10%的助力，中期30%，后期系统反而可能成为限制"
    cost_mechanism: "每次使用系统必须付出有意义的代价，代价随能力增长而加重"
    failure_mode: "系统必须有失灵/被干扰/被反噬的可能性"

  step3_blacklist:
    banned_systems:
      - "签到系统（每天签到获得奖励）— 太被动，缺乏戏剧性"
      - "鉴定系统（看一眼就知道属性）— 太万能，消除悬念"
      - "无限抽奖/盲盒系统 — 纯靠运气，缺乏策略性"
      - "直接复制他人能力的系统 — 消除角色独特性"
      - "任何能直接'毁灭世界/秒杀一切'的系统 — 破坏叙事张力"
    banned_patterns:
      - "系统任务 = 主线剧情（系统不应该替代作者安排剧情）"
      - "系统奖励直接解决当前危机（太巧合，缺乏说服力）"
      - "系统升级 = 角色成长（角色的内在成长必须独立于系统）"

  step4_originality_test:
    - "用一句话描述系统核心机制"
    - "在脑中搜索：是否在任何已知网文中见过类似设定？"
    - "如果答案是'是'或'不确定'，必须进一步变异直到确信独特"
    - "最终系统描述应该让人反应：'这个角度我没想到过'"

  good_examples:
    - "情绪交易系统：主角可以将自己的情绪（快乐、悲伤、恐惧）具象化交易，买入他人情绪获得对应能力，但长期缺失某种情绪会导致人格异变"
    - "因果审计系统：主角能看到每个人身上的'因果账本'，可以通过'审计'找出因果漏洞并利用，但每次审计都会让自己的因果变得更复杂"
    - "生态位系统：主角在一个类似自然生态的能力体系中，必须找到自己的'生态位'——不是最强，而是最不可替代"

  forbidden:
    - ❌ 系统在第一章就给主角开满级权限
    - ❌ 系统没有任何使用限制或代价
    - ❌ 系统的存在没有世界观层面的解释（哪怕是悬念式的）
    - ❌ 主角完全依赖系统，离开系统就是废物
```

### 引擎 D：爽文节奏控制系统

**核心原则：爽感不是一条直线，而是一条不断攀升的波浪线。低谷是为了让高峰更高。**

```yaml
writing_engine: "thrill_curve"
priorities:
  - 节奏感 > 单次爽度 > 剧情深度
  - 每个"爽点"必须比上一个更强或更意外
  - 低谷期不是无聊期，而是"蓄力期"

thrill_curve_model:
  name: "潮汐爽感模型"
  description: |
    将全书划分为多个"潮汐周期"，每个周期包含：
    蓄力（20%）→ 小爽（10%）→ 压制（30%）→ 反转（10%）→ 大爽（20%）→ 余韵（10%）

  phase_definitions:
    蓄力期:
      purpose: "建立期待，让读者知道'好事要来了'"
      techniques:
        - "展示主角获得新能力/资源但尚未使用"
        - "铺垫对手的强大，制造'怎么赢'的悬念"
        - "暗示即将到来的机遇（比赛/秘境/拍卖会）"
      duration: "2-4章"
      reader_emotion: "期待、好奇"

    小爽期:
      purpose: "给读者一个'开胃菜'，证明主角确实在变强"
      techniques:
        - "碾压一个之前棘手的小角色"
        - "展示新能力的冰山一角"
        - "获得旁人的惊叹/认可"
      duration: "1-2章"
      reader_emotion: "小满足、想看更多"

    压制期:
      purpose: "制造逆境，把爽感的弹簧压到最低"
      techniques:
        - "遇到更强的对手，之前的优势被克制"
        - "遭遇背叛/误解/陷害"
        - "付出代价/失去重要的东西"
        - "被逼入绝境，常规手段全部失效"
      duration: "4-8章"
      reader_emotion: "焦虑、愤怒、'主角快反击啊！'"
      critical_rule: "压制必须合理，不能为了压而压。对手的优势要有逻辑支撑。"

    反转期:
      purpose: "打破僵局的关键一击，通常是意料之外情理之中"
      techniques:
        - "之前埋的伏笔在此刻发挥作用"
        - "主角的'弱点'反而成为破局关键"
        - "意想不到的盟友/信息出现"
        - "主角突破自我限制（心理层面，不是简单的升级）"
      duration: "1-2章"
      reader_emotion: "惊喜、恍然大悟、'原来如此！'"

    大爽期:
      purpose: "全力释放积累的爽感，让读者获得最大满足"
      techniques:
        - "以压倒性的方式击败对手（但过程要有层次，不是一拳秒杀）"
        - "打脸环节：之前看不起主角的人目瞪口呆"
        - "连锁反应：一次胜利带来多重收益（名声/资源/关系）"
        - "揭示主角一直隐藏的底牌"
      duration: "3-5章"
      reader_emotion: "极度满足、想分享给别人看"
      critical_rule: "大爽不等于无脑碾压。最佳爽感 = 实力碾压 + 智商碾压 + 气势碾压 的组合"

    余韵期:
      purpose: "消化爽感，同时为下一个周期埋种子"
      techniques:
        - "展示胜利的后续影响（敌人的反应、盟友的态度变化）"
        - "主角短暂的平静/反思"
        - "暗示更大的挑战即将到来"
      duration: "1-2章"
      reader_emotion: "满足但隐隐期待"

  escalation_rule: |
    每个潮汐周期的"大爽峰值"必须高于上一个周期。
    实现方式：
    - 对手层级递增（同辈 → 长辈 → 势力 → 国家 → 世界级）
    - 赌注递增（面子 → 资源 → 生死 → 所爱之人 → 世界存亡）
    - 主角展示的能力层级递增
    - 打脸的观众规模递增（个人 → 小团体 → 大型场合 → 天下皆知）

  anti_patterns:
    - name: "一路碾压综合症"
      symptom: "主角从头到尾没有遇到真正的威胁"
      fix: "每个周期的压制期必须让主角真正陷入困境，至少有1次让读者觉得'这次可能真的输了'"
    - name: "假压制"
      symptom: "压制期只持续1章就反转，读者感受不到压力"
      fix: "压制期至少4章，且要有实质性的损失（不只是受伤，还要失去资源/关系/信任）"
    - name: "爽感通胀"
      symptom: "每次都是同样的打脸套路，读者麻木"
      fix: "每个周期的爽点类型必须不同（武力碾压/智谋胜出/以弱胜强/绝境翻盘/身份揭示...）"
    - name: "无意义低谷"
      symptom: "低谷期只是在拖字数，没有推进角色成长或剧情"
      fix: "低谷期必须完成至少1项：角色内在成长/关系深化/伏笔铺设/世界观扩展"

  forbidden:
    - ❌ 连续2个周期使用相同的爽点模式
    - ❌ 压制期没有合理原因（不能是主角突然变蠢）
    - ❌ 大爽期靠外挂/天降神兵解决（必须与主角的努力/智慧相关）
    - ❌ 全书没有一次让主角真正失败/付出重大代价的情节
    - ❌ 打脸对象全是脸谱化的蠢货（对手也要有智商和尊严）
```

```markdown

根据以下参数，生成一部长篇网文小说的完整基础设定。

## 约束
- 分卷规划至少覆盖全书 60% 的内容
- 每卷必须有独立的小高潮和悬念钩子
- 世界观设定要自洽，避免逻辑矛盾
- 如果是100万字以上，至少规划5个大卷
- 【新增】必须输出 innovation_report，说明本作的创新性

4. 描写要求：
   - 动作场面：短句为主，节奏紧凑
   - 情感场面：细腻内心独白 + 环境烘托
   - 日常场面：轻松幽默，穿插信息

### 潮汐阶段专属写法
根据当前所处的潮汐阶段，调整写作策略：

**蓄力期写法：**
- 节奏偏慢，注重氛围营造和信息铺垫
- 通过角色对话/内心活动暗示即将到来的挑战
- 可以穿插轻松日常，但必须包含至少1个"不经意的伏笔"
- 结尾留下"暴风雨前的宁静"感

**小爽期写法：**
- 节奏明快，给读者一个小甜头
- 展示主角的部分实力，但保留底牌
- 旁观者的惊讶反应要自然，不要过度吹捧
- 点到为止，让读者"还没看够"

**压制期写法：**
- 节奏紧凑但沉重，制造窒息感
- 主角的困境必须真实可信，不能靠主角降智实现
- 允许主角犯错，但错误必须符合其性格和当时的信息量
- 配角的反应要真实（有人帮忙、有人旁观、有人落井下石）
- 每章结尾的情况要比开头更糟，形成"下坠感"

**反转期写法：**
- 节奏突变，从压抑转向爆发
- 反转的触发点必须在前文有铺垫（可以很隐蔽）
- 用短句和快节奏推进，制造"终于等到了"的释放感
- 这是全周期最关键的1-2章，要写得精彩

**大爽期写法：**
- 全力输出，节奏酣畅淋漓
- 打脸/碾压/逆袭的过程要有层次（不是一拳解决）
- 多角度展示爽感：主角视角的从容 + 对手视角的震惊 + 旁观者视角的惊叹
- 对话要有记忆点，写出"名场面"感
- 适当穿插之前压制期的回忆闪回，形成强烈对比

**余韵期写法：**
- 节奏放缓，让读者"回味"
- 展示胜利的连锁反应和后续影响
- 角色之间的关系因此事件产生微妙变化
- 结尾暗示新的挑战/阴谋，为下一个周期蓄力

### 类型专属写法

**规则怪谈类额外规范：**
- 规则描述必须精确，用词不能模棱两可
- 角色推理过程要写出来，让读者能跟着一起思考
- 恐怖氛围通过"细节异常"营造，而非直接描写血腥
- 每次有角色遇难，必须在前文能找到"他违反了什么"
- 主角的求生策略必须基于已知信息，禁止开挂

**玄幻/科幻类额外规范：**
- 每章至少有1个让读者感到"新奇"的细节（可以是世界观、能力、生物、文化）
- 战斗描写不能只是"你一拳我一剑"，要体现能力体系的独特性
- 升级/突破场景要有仪式感，不能草草带过
- 新设定的引入要自然，避免大段"百科全书式"解说

**系统流额外规范：**
- 系统界面/提示的展示要简洁，不要占太多篇幅
- 主角使用系统时必须有策略性思考，不能无脑点确认
- 系统的代价要在正文中体现出来（不只是数值扣减，要有叙事层面的影响）
- 主角脱离系统时也要有独立解决问题的能力

**爽文额外规范：**
- 打脸对象必须先建立其"值得被打脸"的理由（傲慢、欺压、不公）
- 围观群众的反应要分层次（有人震惊、有人不信、有人嫉妒、有人佩服）
- 爽感释放后要有"冷却"，不能连续3章以上持续高强度爽
- 主角的强大要有合理来源，不能让读者觉得"凭什么"

### 系统流专项检查（仅 system_innovation 引擎）

23. ☐ 系统功能是否与之前章节一致（无突然新增未解释的功能）

### 玄幻/科幻专项检查（仅 imagination_first 引擎）
24. ☐ 本章是否有至少1个新奇的世界观/能力细节

## 设计说明

| 设计决策 | 理由 |
|---------|------|
| JSON 结构化输出 | 方便机器解析和人工查阅，确保数据一致性 |
| 章节后记机制 | 解决 LLM 上下文窗口限制，用结构化摘要代替全文记忆 |
| 角色状态快照 | 每章只加载出场角色信息，减少 token 消耗 |
| 伏笔追踪系统 | 防止长篇小说常见的"挖坑不填"问题 |
| 连贯性检查器 | 自动化质量保障，降低人工审校负担 |
| 分阶段执行 | 避免单次提示过长导致输出质量下降 |
| **v2.0 新增** | |
| 类型差异化引擎 | 不同类型有本质不同的写作逻辑，通用规则无法覆盖 |
| 三层规则体系（规则怪谈） | 规则类小说的恐惧来源于逻辑，模糊的规则=失败的恐怖 |
| 逆向设定法（玄幻/科幻） | 主动规避套路，用系统化方法保证想象力的新颖性 |
| 领域迁移法（系统流） | 从非游戏领域获取灵感，避免系统设计雷同 |
| 潮汐爽感模型 | 爽感需要对比才有感觉，一路碾压=一路平淡 |
| 系统助力占比追踪 | 防止主角沦为系统的提线木偶，保持角色主体性 |
| 爽点类型轮换机制 | 防止读者对同一种爽感模式产生审美疲劳 |
