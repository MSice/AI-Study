# 长篇网文创作系统 v4.0 — 提示词模板

> 一套模块化的提示词系统，用于驱动 LLM 自动生成并持续创作长篇网文小说。
> v4.0 核心升级：**三阶段门禁 + 用户预览确认**，确保方向正确后再投入后续工作。

---

## v4.0 核心理念

**先定方向，再动笔。方向错了，写再多也是浪费。**

v3.0 的流程是线性流水线——生成完就往下走。v4.0 将全流程重构为三大阶段，
每个阶段内部自动执行，但阶段之间必须经过 **评分门禁 + 用户预览确认** 才能推进：

```
┌─────────────────────────────────────────────────────────────────────┐
│  第一阶段：基建锁定                                                  │
│  世界观 → 角色 → 关系 → 故事概览                                     │
│  ──→ 基建整体评分 ≥95 ──→ 【用户预览确认】──→ 锁定基建               │
├─────────────────────────────────────────────────────────────────────┤
│  第二阶段：开篇预览                                                  │
│  详细大纲 → 黄金三章正文                                             │
│  ──→ 大纲评分 ≥90 ──→ 开篇评分 ≥90 ──→ 【用户预览确认】──→ 开篇定稿  │
├─────────────────────────────────────────────────────────────────────┤
│  第三阶段：连载生产                                                  │
│  第4章起循环写作                                                     │
│  ──→ 每章评分 ≥80 ──→ 通过则继续，不通过则修改                       │
└─────────────────────────────────────────────────────────────────────┘
```

**关键规则：**
- 第一阶段未锁定，不得进入第二阶段
- 第二阶段未定稿，不得进入第三阶段
- 已锁定的基建，后续章节不得违反

## 第一阶段：基建锁定

> 目标：确定"这本书长什么样"——世界观、角色、关系、故事走向。
> 这是全书的地基，必须打牢后才能动笔。

### 执行步骤

```
1. 阅读 [00-平台适配与字数分段](./00-平台适配与字数分段.md) → 确认平台策略和字数档位
2. 阅读 [01-类型写作引擎](./01-类型写作引擎.md) → 确认激活哪些引擎
3. 执行 [03-五阶段核心提示词](./03-五阶段核心提示词.md) 的阶段一（世界观）
4. 执行阶段二（角色档案）
5. 执行阶段三（关系网络）
6. 生成【故事概览卡】（自动汇总，见下方格式）
```

### 故事概览卡

阶段一~三全部完成后，系统自动生成一张"故事概览卡"供用户预览：

```markdown
## 📖 故事概览卡

**书名**：《xxx》
**类型**：xxx | **平台**：xxx | **字数**：xxx万字 | **基调**：xxx

### 一句话简介
xxx

### 世界观速览
- 时代背景：xxx
- 力量体系：xxx（创新点：xxx）
- 核心规则：xxx

### 主要角色
| 角色 | 身份 | 性格关键词 | 核心驱动力 | 与主角关系 |
|------|------|-----------|-----------|-----------|
| xxx  | 主角 | xxx       | xxx       | —         |
| xxx  | xxx  | xxx       | xxx       | xxx       |

### 核心关系线
- xxx 与 xxx：xxx（将如何演变：xxx）
- 感情线：xxx

### 故事主线
- 核心矛盾：xxx
- 主角目标：xxx
- 分卷规划摘要：
  - 第一卷：xxx（高潮：xxx）
  - 第二卷：xxx（高潮：xxx）
  - ...

### 本作创新点
xxx
```

### 评分与确认

```
故事概览卡生成后：

1. 运行【基建整体评分】（见 05-质量评分体系.md）
   → 世界观 + 角色 + 关系 + 故事弧 作为一个整体评分
   → 满分100分，≥95分通过

2. 评分通过后，将故事概览卡呈现给用户预览
   → 用户确认："这个方向我满意，锁定基建"
   → 或用户反馈："xxx 需要调整"→ 修改后重新评分

3. 用户确认后，基建正式锁定
   → 后续所有章节不得违反已锁定的设定
   → 如需修改基建，必须重新走第一阶段流程
```

## 第二阶段：开篇预览

> 目标：让用户看到"这本书读起来什么感觉"——通过黄金三章的实际正文来验证。
> 开篇是网文的生死线，必须打磨到位后才能进入连载。

### 执行步骤

```
1. 执行 [03-五阶段核心提示词](./03-五阶段核心提示词.md) 的阶段四（章节大纲）
   → 先生成前10-20章的详细大纲
   → 运行【大纲评分】≥90分通过

2. 执行 [02-黄金三章与开篇设计](./02-黄金三章与开篇设计.md)
   → 撰写前3章（或根据类型调整：起点/悬疑/历史为前5章，知乎为前2章）
   → 运行【开篇评分】≥90分通过

3. 评分通过后，将开篇正文呈现给用户预览
   → 用户确认："开篇我满意，定稿"
   → 或用户反馈："节奏太慢/角色不够鲜明/钩子不够强..."→ 修改后重新评分

4. 用户确认后，开篇正式定稿
   → 进入第三阶段连载生产
```

---

## 第三阶段：连载生产

> 目标：稳定高质量输出后续章节。
> 基建已锁定、开篇已定稿，此阶段专注于"写好每一章"。
>
> **v4.0 关键特性**：用户指定每次生成的数量，批次完成后预览、修改，满意后再继续。

### 步骤一：用户指定生成任务

每次进入连载生产前，用户先指定本批次要生成多少内容：

```yaml
batch_config:
  mode: ""          # 三选一：by_chapters / by_volume / to_milestone
  
  # 模式一：按章数
  by_chapters:
    count: 5        # 本次生成5章（用户自定义）
  
  # 模式二：按卷
  by_volume:
    volume: 2       # 生成第2卷全部章节
  
  # 模式三：到指定节点
  to_milestone:
    target: "主角突破金丹期"  # 写到这个剧情节点为止
```

**示例：**
- "帮我续写5章" → `mode: by_chapters, count: 5`
- "把第二卷写完" → `mode: by_volume, volume: 2`
- "写到主角和反派第一次正面交锋" → `mode: to_milestone, target: "主角与反派正面交锋"`

### 步骤二：批量生成 + 自动评分

系统根据用户指定的数量，逐章生成并自动评分：

```
批次开始（共N章）
    │
    ▼
┌─────────────────────────────────────────────┐
│  写第 i 章（执行阶段五）                      │
│      │                                       │
│      ▼                                       │
│  状态更新器（更新角色/关系数据）               │
│      │                                       │
│      ▼                                       │
│  连贯性检查器（34项检查）                     │
│      │                                       │
│      ▼                                       │
│  爽感审计器（5维度量化）                      │
│      │                                       │
│      ▼                                       │
│  章节评分 ≥80?                               │
│      │                                       │
│     ╱ ╲                                      │
│   是    否 → 自动修改后重评                   │
│    │                                         │
│    ▼                                         │
│  i < N? ── 是 → 写第 i+1 章（循环）          │
│    │                                         │
│   否                                         │
└────┼────────────────────────────────────────┘
     ▼
  批次完成，进入用户预览
```

### 步骤三：批次预览 + 用户修改

批次全部章节生成完毕后，呈现给用户预览：

```markdown
## 📋 批次生成报告

**本批次**：第4-8章（共5章）
**总字数**：约12,000字
**平均评分**：87.4分

| 章节 | 标题 | 字数 | 评分 | 等级 | 潮汐阶段 | 备注 |
|------|------|------|------|------|---------|------|
| 第4章 | xxx | 2400 | 89 | B+ | 蓄力期 | — |
| 第5章 | xxx | 2300 | 85 | B  | 小爽期 | — |
| 第6章 | xxx | 2500 | 91 | A  | 压制期 | 本批最佳 |
| 第7章 | xxx | 2200 | 82 | B  | 压制期 | 钩子偏弱 |
| 第8章 | xxx | 2600 | 90 | A  | 反转期 | — |

### 各章摘要
- **第4章**：xxx（一句话摘要）
- **第5章**：xxx
- ...
```

用户预览后有以下选择：

```yaml
batch_review:
  option_1:
    action: "全部确认"
    description: "所有章节都满意，确认后指定下一批"
    result: "本批次定稿，进入下一批生成任务指定"
  
  option_2:
    action: "修改指定章节"
    description: "某几章需要修改"
    example: "第7章的结尾钩子不够强，帮我改一下"
    result: "修改指定章节 → 重新评分 → 更新批次报告 → 再次预览"
  
  option_3:
    action: "从某章开始重写"
    description: "某章之后的走向不对，从那章开始重写"
    example: "第6章开始剧情走偏了，从第6章重新写"
    result: "丢弃第6章及之后的章节 → 从第6章重新生成"
  
  option_4:
    action: "暂停连载"
    description: "本次到此为止，下次再继续"
    result: "保存当前进度，记录断点"
```

### 特殊节点

- **每10章**：重新评估大纲，确认后续方向
- **每卷结尾**：进行"卷末审计"，检查本卷整体质量和下卷衔接
- **长篇/超长篇每50章**：角色状态全面审计，防止角色遗忘或设定漂移
- **批次跨越特殊节点时**：在该节点处自动暂停，先完成审计再继续
