# 01 — 类型写作引擎（8大引擎）

> 每种小说类型有本质不同的写作逻辑。本模块定义了8大类型引擎，
> 阶段一填入类型后系统会自动激活对应引擎，贯穿全流程。
> v3.0 在 v2.0 的 A-D 引擎基础上，新增 E-H 四个引擎。

---

## 引擎总览

| 引擎 | 类型 | 核心原则 | 来源版本 |
|------|------|---------|---------|
| A | 鬼怪规则类 | 逻辑即恐惧 | v2.0 |
| B | 玄幻/科幻类 | 想象力第一 | v2.0 |
| C | 系统流/穿越类 | 系统是工具不是万能钥匙 | v2.0 |
| D | 爽文节奏控制 | 爽感是波浪线不是直线 | v2.0 |
| **E** | **言情类** | **情感真实 > 情节离奇** | **v3.0 新增** |
| **F** | **悬疑推理类** | **公平博弈，线索先行** | **v3.0 新增** |
| **G** | **都市类** | **代入感即爽感** | **v3.0 新增** |
| **H** | **历史架空类** | **历史理性 + 人物多面性** | **v3.0 新增** |

## 引擎 A：鬼怪规则类（恐怖/灵异/无限流/规则怪谈）

**核心原则：逻辑即恐惧。规则越清晰，违反规则的后果越恐怖。**

```yaml
writing_engine: "rule_horror"
priorities:
  - 逻辑自洽 > 氛围渲染 > 想象力
  - 规则必须可验证：读者能根据已知规则推导出"谁会死"
  - 恐惧来源于"已知的确定性"而非"未知的模糊感"

rule_system_design:
  structure: "三层规则体系"
  layers:
    - name: "表层规则"
      desc: "角色和读者最先发现的规则，通常不完整或有误导性"
      example: "晚上12点不能照镜子"
      reveal_timing: "前3章内展示"
    - name: "深层规则"
      desc: "隐藏在表层之下的真实运作逻辑"
      example: "镜子不是禁忌本身，而是'被注视'才会触发诅咒"
      reveal_timing: "中期反转时揭示"
    - name: "元规则"
      desc: "整个怪谈世界的底层法则，解释为什么规则存在"
      example: "所有规则都是某个意识体的'游戏条款'"
      reveal_timing: "大结局或关键转折"

  rule_count: "5-9条为宜，过多造成读者审美疲劳且易产生逻辑漏洞"
  
  faction_design:
    - "正方立场（人类）：通过制定规则进行保护"
    - "反方立场（异常）：通过干扰规则（矛盾规则、污染等）进行渗透"
    - "新手不建议设置多立场，容易自相矛盾"

  logic_constraints:
    - 每条规则必须有明确的触发条件、生效范围、违反后果
    - 规则之间不能矛盾，但可以存在"看似矛盾实则不同层级"的情况
    - 主角破解规则的方式必须基于已给出的线索，禁止凭空开挂
    - 每次有角色死亡/遭遇危险，读者回溯时必须能找到"他违反了哪条规则"
    - 遵守规则必须能活下来，但不一定能找到真相
    - 善用比喻/借喻润色规则，将真实故事元素抽象化

  tension_formula:
    - "信息差制造悬念：读者知道规则但角色不知道 → 紧张感"
    - "规则冲突制造绝境：遵守A规则就必然违反B规则 → 两难"
    - "规则迭代制造升级：旧规则被破解，新规则更致命 → 层层加码"

  chapter_rhythm:
    - "规则发现章 → 规则试探章 → 规则验证章（有人因此遇难）→ 规则破解章 → 新规则出现"
    - 每5-8章完成一个"规则循环"
    - 禁止连续3章以上没有规则相关推进

  forbidden:
    - ❌ 规则模糊不清，无法被读者理解和推导
    - ❌ 主角靠"灵光一闪"解决问题，没有逻辑铺垫
    - ❌ 鬼怪的行为前后不一致（除非有更高层规则解释）
    - ❌ 恐怖场景只靠jump scare式描写，没有规则支撑
    - ❌ 角色明知规则还犯蠢（除非有合理的心理/情境压力解释）
    - ❌ 宣扬不良价值观，文风应积极向上
```

## 引擎 B：玄幻 / 科幻类（仙侠/奇幻/星际/赛博朋克）

**核心原则：想象力是第一生产力。设定必须"从未见过"但"细想合理"。**

```yaml
writing_engine: "imagination_first"
priorities:
  - 新颖性 > 合理性 > 逻辑严密性
  - 设定要让读者产生"卧槽还能这样"的惊叹感
  - 合理性是底线，不是天花板

worldbuilding_innovation:
  method: "逆向设定法"
  steps:
    - "列出该类型最常见的10个设定套路"
    - "逐一反转或融合，生成候选设定"
    - "用'三秒钟测试'筛选：读者看到设定的前三秒是否会产生好奇心"
    - "用'深挖测试'验证：这个设定能否支撑至少3个独立的剧情冲突"

  trending_innovations:
    - "赛博修仙：科技与修真的跨界融合"
    - "量子武道：微观物理与武学的结合"
    - "传统修真的现代化重构"
    - "社会结构与修炼体系深度匹配"

  innovation_checklist:
    - ☐ 力量体系是否有独特的获取/升级方式（不是简单的打怪升级）
    - ☐ 世界观是否有至少1个"反直觉但自洽"的核心设定
    - ☐ 种族/阵营划分是否超越了"人类vs异族"的二元对立
    - ☐ 科技/法术的代价体系是否有创意
    - ☐ 地理/空间设定是否有独特的物理法则或生态逻辑

  anti_cliche_rules:
    - "如果一个设定在起点排名前100的同类小说中出现过3次以上，必须变体"
    - "修炼体系禁止使用：炼气→筑基→金丹→元婴 这类标准模板，除非有本质性改造"
    - "科幻设定禁止使用：人类联邦vs外星帝国 这类标准框架，除非有独特的政治/文化维度"

  imagination_amplifiers:
    - "跨领域嫁接：将现实中某个冷门领域的逻辑移植到奇幻世界"
    - "尺度跳跃：在微观/宏观之间切换视角，制造认知冲击"
    - "规则颠覆：让读者以为理解了世界规则，然后揭示更大的框架"

  writing_style:
    - "战斗描写要体现能力体系的独特性，不是通用打斗"
    - "升级/突破场景要有仪式感，不能草草带过"
    - "新设定的引入要自然，避免大段'百科全书式'解说"
    - "每章至少有1个让读者感到新奇的细节"

  forbidden:
    - ❌ 力量体系只是换了名字的"打怪升级"
    - ❌ 世界观是"中世纪欧洲+魔法"或"古代中国+修仙"的简单叠加
    - ❌ 外星文明只是"长得不同的人类"
    - ❌ 科技设定没有社会影响
```

## 引擎 C：系统流 / 穿越类

**核心原则：系统是工具不是万能钥匙。系统的独特性决定小说的独特性。**

```yaml
writing_engine: "system_innovation"
priorities:
  - 系统创新性 > 爽感 > 合理性
  - 系统必须有"从未在网文中出现过"的核心机制
  - 系统强大但有明确边界，不能解决所有问题

system_design_protocol:
  step1_innovation_generation:
    method: "领域迁移法"
    process:
      - "从现实世界选择一个非游戏领域（拍卖、烹饪、社交信用、天气预报、股票交易、考古、快递物流...）"
      - "提取该领域的核心机制"
      - "将机制转化为系统功能"
      - "设计系统的独特交互方式（不是简单的面板+任务+奖励）"

  step2_balance_check:
    power_ceiling: "系统的最强能力不能超过当前世界观天花板的70%"
    growth_curve: "前期系统提供10%的助力，中期30%，后期系统反而可能成为限制"
    cost_mechanism: "每次使用系统必须付出有意义的代价，代价随能力增长而加重"
    failure_mode: "系统必须有失灵/被干扰/被反噬的可能性"

  step3_blacklist:
    banned_systems:
      - "签到系统（每天签到获得奖励）— 太被动，缺乏戏剧性"
      - "鉴定系统（看一眼就知道属性）— 太万能，消除悬念"
      - "无限抽奖/盲盒系统 — 纯靠运气，缺乏策略性"
      - "直接复制他人能力的系统 — 消除角色独特性"
      - "任何能直接'毁灭世界/秒杀一切'的系统 — 破坏叙事张力"
    banned_patterns:
      - "系统任务 = 主线剧情（系统不应该替代作者安排剧情）"
      - "系统奖励直接解决当前危机（太巧合，缺乏说服力）"
      - "系统升级 = 角色成长（角色的内在成长必须独立于系统）"

  step4_originality_test:
    - "用一句话描述系统核心机制"
    - "搜索：是否在任何已知网文中见过类似设定？"
    - "如果答案是'是'或'不确定'，必须进一步变异直到确信独特"

  good_examples:
    - "情绪交易系统：将情绪具象化交易，买入他人情绪获得对应能力，但长期缺失某种情绪会导致人格异变"
    - "因果审计系统：看到每个人身上的'因果账本'，通过'审计'找出因果漏洞并利用"
    - "生态位系统：在类似自然生态的能力体系中找到自己的'生态位'——不是最强，而是最不可替代"

  forbidden:
    - ❌ 系统在第一章就给主角开满级权限
    - ❌ 系统没有任何使用限制或代价
    - ❌ 系统的存在没有世界观层面的解释
    - ❌ 主角完全依赖系统，离开系统就是废物
```

## 引擎 D：爽文节奏控制系统

**核心原则：爽感不是一条直线，而是一条不断攀升的波浪线。低谷是为了让高峰更高。**

```yaml
writing_engine: "thrill_curve"
priorities:
  - 节奏感 > 单次爽度 > 剧情深度
  - 每个"爽点"必须比上一个更强或更意外
  - 低谷期不是无聊期，而是"蓄力期"

thrill_curve_model:
  name: "潮汐爽感模型"
  formula: "蓄力(20%) → 小爽(10%) → 压制(30%) → 反转(10%) → 大爽(20%) → 余韵(10%)"

  phase_definitions:
    蓄力期:
      purpose: "建立期待"
      techniques: ["展示新能力未使用", "铺垫对手强大", "暗示即将到来的机遇"]
      duration: "2-4章"
      reader_emotion: "期待、好奇"

    小爽期:
      purpose: "开胃菜"
      techniques: ["碾压之前棘手的小角色", "展示新能力冰山一角", "获得旁人惊叹"]
      duration: "1-2章"
      reader_emotion: "小满足、想看更多"

    压制期:
      purpose: "把弹簧压到最低"
      techniques: ["更强对手克制优势", "遭遇背叛/陷害", "付出代价/失去重要的东西", "绝境"]
      duration: "4-8章"
      reader_emotion: "焦虑、愤怒、'主角快反击啊！'"
      critical_rule: "压制必须合理，不能为了压而压"

    反转期:
      purpose: "意料之外情理之中的破局"
      techniques: ["伏笔发挥作用", "弱点反成关键", "意外盟友/信息", "突破自我限制"]
      duration: "1-2章"
      reader_emotion: "惊喜、恍然大悟"

    大爽期:
      purpose: "全力释放积累的爽感"
      techniques: ["压倒性击败（有层次）", "打脸", "连锁收益", "揭示底牌"]
      duration: "3-5章"
      reader_emotion: "极度满足"
      critical_rule: "最佳爽感 = 实力碾压 + 智商碾压 + 气势碾压 的组合"

    余韵期:
      purpose: "消化爽感，埋下一周期种子"
      techniques: ["展示后续影响", "短暂平静/反思", "暗示新挑战"]
      duration: "1-2章"
      reader_emotion: "满足但隐隐期待"

  escalation_rule: |
    每个潮汐周期的"大爽峰值"必须高于上一个周期。
    对手层级递增（同辈→长辈→势力→国家→世界级）
    赌注递增（面子→资源→生死→所爱之人→世界存亡）
    打脸观众规模递增（个人→小团体→大型场合→天下皆知）

  anti_patterns:
    - name: "一路碾压综合症"
      fix: "每个周期必须让主角真正陷入困境"
    - name: "假压制"
      fix: "压制期至少4章，且要有实质性损失"
    - name: "爽感通胀"
      fix: "每个周期的爽点类型必须不同"
    - name: "无意义低谷"
      fix: "低谷期必须完成：角色成长/关系深化/伏笔铺设/世界观扩展"

  番茄适配:
    - "每2000字需设置小高潮或反转（番茄特殊要求）"
    - "每3-5千字一个小高潮，每2-3万字一个大打脸"
    - "每万字需要一个爽点"

  forbidden:
    - ❌ 连续2个周期使用相同的爽点模式
    - ❌ 压制期没有合理原因（不能是主角突然变蠢）
    - ❌ 大爽期靠外挂/天降神兵解决
    - ❌ 全书没有一次让主角真正失败/付出重大代价
    - ❌ 打脸对象全是脸谱化的蠢货
```

## 引擎 E：言情类（甜宠/虐恋/双强/古言/现言）【v3.0 新增】

**核心原则：情感真实 > 情节离奇。读者要的不是完美爱情，而是"真实的心动"。**

```yaml
writing_engine: "romance"
priorities:
  - 情感真实性 > 情节戏剧性 > 世界观复杂度
  - CP感是生命线，两个人在一起必须让读者觉得"就该是他们"
  - 感情发展要有节奏，不能一见钟情也不能拖到天荒地老

cp_design:
  method: "互补碰撞法"
  steps:
    - "先确定一个角色的核心性格"
    - "设计另一个角色时考虑：相似点（共鸣基础）+ 不同点（吸引力）+ 矛盾点（冲突来源）"
    - "两人的弱点必须恰好是对方能弥补的"
    - "设计只属于两个人才知道的'秘密/暗号/习惯'，强化CP专属感"

  cp_types:
    甜宠:
      core: "制造期待 → 延迟满足 → 满足期待 的循环"
      techniques:
        - "用第三方视角描写'甜与宠'，避免当事人视角显得油腻"
        - "在低谷时适当加糖（'玻璃渣中扣糖'），让甜度更浓"
        - "反差感人设（表面高冷实际宠溺/表面软萌实际腹黑）"
        - "日常互动比大事件更能制造心动感"
      pacing: "甜7虐3，虐的部分是误会/外部阻力，不是互相伤害"

    虐恋:
      core: "先撒糖再揭露残酷真相，制造反差冲击"
      techniques:
        - "矛盾由角色出发点不同造成，不是无脑误会"
        - "用场景氛围烘托情绪（潮湿冷硬的感官细节）"
        - "对话中藏潜台词，用'口是心非'体现压抑情感"
        - "'三明治法'：糖→刀子→糖，让读者在希望和绝望间反复"
        - "虐点必须合理：逻辑上的虐（事件困境）+ 感情上的虐（为爱牺牲）"
      pacing: "甜3虐7，但每次虐完必须有一个微小的希望"

    双强:
      core: "两个角色都有独立人格与实力，互补而非单向依赖"
      techniques:
        - "各自有独立的事业线/成长线"
        - "在各自领域都是顶尖，但在感情面前都有笨拙的一面"
        - "合作时1+1>2，分开时各自也能独当一面"
        - "冲突来自理念/立场差异，而非实力碾压"
      pacing: "事业线和感情线交替推进，比例约5:5"

  emotion_rhythm:
    name: "心跳曲线模型"
    phases:
      - "初遇期：好奇/排斥 → 第一次心动（不自知）"
      - "试探期：暧昧/拉扯 → 确认心意（但有阻碍）"
      - "热恋期：甜蜜/磨合 → 感情深化"
      - "危机期：误会/分离/外部威胁 → 感情考验"
      - "回归期：和解/重逢 → 感情升华"
    rule: "每个阶段都要有'心动瞬间'——一个让读者截图分享的名场面"

  female_character_rules:
    - "女性角色必须有独立人格和目标，不能只围着男主转"
    - "女主的魅力不能只靠外貌，要有能力/智慧/性格上的闪光点"
    - "女配不能只是'恶毒女配'模板，要有自己的动机和逻辑"

  forbidden:
    - ❌ 强制爱/囚禁/PUA 被美化为浪漫
    - ❌ 女主除了谈恋爱没有任何其他追求
    - ❌ 感情线全靠误会推动（误会要有合理信息差支撑）
    - ❌ 男主的"霸道"越过人身自由的边界
    - ❌ 配角只为拆CP而存在，没有自己的逻辑
    - ❌ 全程甜到发腻没有任何波折（甜宠也需要节奏）
```

## 引擎 F：悬疑推理类（本格推理/社会派/心理悬疑）【v3.0 新增】

**核心原则：公平博弈，线索先行。读者在真相揭晓前必须有机会自己推导出答案。**

```yaml
writing_engine: "mystery"
priorities:
  - 逻辑严密性 > 氛围营造 > 角色深度
  - 所有关键线索必须在揭秘前呈现给读者（公平原则）
  - 真相必须"意料之外，情理之中"

mystery_design:
  clue_system:
    rule: "三线索原则"
    description: |
      每个关键真相至少通过3条独立线索暗示：
      - 线索A：明显但容易被忽略的细节
      - 线索B：隐藏在对话或行为中的暗示
      - 线索C：需要结合其他信息才能理解的间接证据
    timing: "所有关键线索必须在揭秘章的至少3章前出现"

  misdirection:
    - "红鲱鱼（误导线索）：至少设置2条，但事后回看必须能发现它们的破绽"
    - "可靠叙述者的不可靠：利用视角限制制造信息差"
    - "时间线错位：非线性叙事制造悬念，但最终时间线必须自洽"

  suspect_design:
    - "至少3个嫌疑人，每个都有动机、机会和手段"
    - "真凶不能是最后才出场的角色（读者会觉得不公平）"
    - "每个嫌疑人都有自己的秘密（不一定与案件相关），增加复杂度"

  reveal_structure:
    - "小揭秘（每5-8章）：揭开一层谜团，但引出更大的疑问"
    - "中揭秘（每卷结尾）：重大真相揭露，颠覆之前的认知"
    - "大揭秘（全书高潮）：所有线索汇聚，真相完整呈现"

  logic_validation:
    - "写完揭秘章后，回溯检查：读者是否有足够信息推导出真相"
    - "检查是否有'作者特权'：主角获得了读者无法获得的信息"
    - "检查时间线：所有事件的发生顺序是否自洽"

  atmosphere_techniques:
    - "环境描写服务于悬疑氛围（光线、声音、温度的异常）"
    - "角色的微表情和小动作暗示内心真实想法"
    - "利用'日常中的违和感'制造不安（而非直接的恐怖描写）"

  forbidden:
    - ❌ 真相依赖读者不可能知道的信息（违反公平原则）
    - ❌ 凶手是最后5章才出现的新角色
    - ❌ 用超自然力量解释本格推理的谜题（除非一开始就设定了超自然世界观）
    - ❌ 侦探/主角靠直觉而非逻辑破案
    - ❌ 线索过于隐晦，正常读者完全无法注意到
    - ❌ 为了制造悬疑而故意隐瞒主角已知的信息（叙述不诚实）
```

## 引擎 G：都市类（重生/赘婿/战神/神医/商战）【v3.0 新增】

**核心原则：代入感即爽感。读者要在主角身上看到"如果我也能这样"的幻想。**

```yaml
writing_engine: "urban_reborn"
priorities:
  - 代入感 > 爽感 > 合理性
  - 身份反差是核心驱动力
  - 现实逻辑为底线，不能太脱离日常认知

identity_reversal_design:
  core: "表面身份 vs 隐藏身份 的巨大反差"
  
  archetypes:
    赘婿/战神:
      setup: "表面废物/被看不起 → 实际身份极其强大"
      key: "被压制越狠，后期反转越爽"
      rhythm: "扮猪吃虎 → 隐忍受辱 → 身份揭露 → 反派跪地"
      innovation: "不能只是'我是XX的儿子'，隐藏身份要有故事性"

    重生:
      setup: "带着前世记忆/经验回到过去"
      key: "第一桶金的获取方式要有新意"
      must_define: ["重生时间点", "主角职业", "第一桶金来源", "前世遗憾"]
      innovation: "不能只是炒股/买房，要结合角色专业背景"

    神医:
      setup: "医术逆袭，前后身份巨大反差"
      key: "医术来源要合理（传承/自学/系统）"
      innovation: "不能每次都是'一针下去病人立刻好了'，要有诊断过程"

    商战:
      setup: "商业智慧碾压对手"
      key: "商业逻辑要经得起基本推敲"
      innovation: "融入真实商业案例的变体，增加可信度"

  face_slapping_design:
    three_act: "挑战认知 → 初步打脸 → 权威崩塌"
    rules:
      - "打脸对象必须先建立'值得被打脸'的理由（傲慢、欺压、不公）"
      - "打脸对象不能全是蠢货，要有聪明但立场对立的对手"
      - "围观群众反应要分层次（震惊/不信/嫉妒/佩服/恐惧）"
      - "打脸规模递增：个人→家族→公司→行业→社会"

  realism_baseline:
    - "涉及法律/医学/商业的情节要有基本常识支撑"
    - "主角的能力来源必须有交代（哪怕是系统/传承）"
    - "反派的行为要符合社会规则（不能动不动就派杀手）"
    - "金钱/权力的获取要有过程，不能凭空出现"

  forbidden:
    - ❌ 主角的隐藏身份没有任何伏笔就突然揭示
    - ❌ 所有女性角色都爱上主角且没有独立人格
    - ❌ 反派智商持续下线只为衬托主角
    - ❌ 涉及违法行为被美化（如暴力讨债、非法经营）
    - ❌ 主角的成功完全靠前世记忆/身份，没有自身努力
```

## 引擎 H：历史架空类（架空历史/种田/宫斗/权谋）【v3.0 新增】

**核心原则：历史理性 + 人物多面性。不是简单开金手指改历史，而是在历史逻辑中寻找可能性。**

```yaml
writing_engine: "historical"
priorities:
  - 历史真实感 > 戏剧性 > 爽感
  - 人物多面性 > 脸谱化善恶
  - 社会运行逻辑 > 个人英雄主义

knowledge_framework:
  layered_research:
    - "第一层：核心史书（建立基本框架）"
    - "第二层：名家著作（吸收专业观点）"
    - "第三层：社科畅销书（跨学科视角）"
    - "第四层：论坛讨论（把握读者认知）"
  tools:
    - "利用历史地图重视地理因素对战争和战略的影响"
    - "了解目标朝代的职官系统、经济制度、社会风俗"

  sub_types:
    架空历史:
      core: "基于历史理性进行推演，不是简单开金手指"
      rules:
        - "保持历史人物基本评价不变，展现多面性"
        - "采用'反向映射'等手法，避免脸谱化"
        - "通过群像互动推动剧情，不是一人独大"
        - "历史进步是多种力量共同作用的结果"
      innovation: "将不断刷新的历史认知与网文技巧融合"

    种田文:
      core: "从贫困到富有的完整过程，细节是灵魂"
      rules:
        - "女主穿越前的专业技能要与种田内容匹配（厨师/农业专家等）"
        - "发家致富过程要有细节：农作物、贩卖、流通钱币等逻辑自洽"
        - "节奏不宜过快或过慢，享受过程"
        - "冲突点：极品亲戚、挣钱困难、与男主家人矛盾"
      pacing: "生活流为主，每5-8章有一次小危机/小突破"

    宫斗/宅斗:
      core: "打败一个个对手的升级过程"
      rules:
        - "对手从弱到强排列，同时拉近与最终BOSS的距离"
        - "女主通常是庶女或被冷落的嫡女（低起点高成长）"
        - "强大的事业线可掩盖薄弱的感情线"
        - "后期可从家族内斗升级到朝廷斗争"
      pacing: "每10-15章解决一个对手，难度递增"

    权谋朝争:
      core: "朝堂博弈，对手是文武百官"
      rules:
        - "利用职官系统赋予角色特性与立场"
        - "政治博弈要有多方势力，不是简单的二元对立"
        - "每次决策都有代价，没有完美选择"
        - "信息战/舆论战/经济战与军事战并重"

  character_design:
    - "历史人物要展现多面性，避免脸谱化"
    - "穿越者的现代知识不能无限制使用（受限于当时的技术条件和社会接受度）"
    - "配角要有自己的立场和利益诉求，不是主角的NPC"

  forbidden:
    - ❌ 穿越者用现代知识无脑碾压古人（古人不是傻子）
    - ❌ 历史人物完全脸谱化（忠臣=好人，奸臣=坏人）
    - ❌ 忽略当时的社会制度和文化背景（如女性地位、阶级观念）
    - ❌ 种田文中的技术发明超越当时的材料和工艺水平
    - ❌ 宫斗中所有对手都是无脑恶毒，没有合理动机
    - ❌ 权谋中的政治博弈过于简单化（现实政治远比小说复杂）
```
