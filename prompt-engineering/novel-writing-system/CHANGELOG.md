## v4.0（2026-03-05）

**主题：三阶段门禁 + 用户预览确认 + 基建锁定机制 + 批量生成控制**

> 核心理念：先定方向，再动笔。方向错了，写再多也是浪费。

### 架构升级：线性流水线 → 三阶段门禁

v3.0 的流程是线性的——每个阶段独立评分后直接进入下一阶段。
v4.0 将全流程重构为三大阶段，每个阶段内部自动执行，
但阶段之间必须经过 **评分门禁 + 用户预览确认** 才能推进。

| 阶段 | 包含内容 | 评分门禁 | 用户确认 | 锁定机制 |
|------|---------|---------|---------|---------|
| **第一阶段：基建锁定** | 世界观+角色+关系+故事概览 | 整体≥95 | 必须 | 🔒 基建锁定 |
| **第二阶段：开篇预览** | 详细大纲+黄金三章 | 大纲≥90+开篇≥90 | 必须 | 📌 开篇定稿 |
| **第三阶段：连载生产** | 第4章起批量写作 | 每章≥80 | 批次完成后 | 批次确认 |

### 核心变更

| 变更项 | v3.0 | v4.0 |
|--------|------|------|
| **基建评分** | 世界观/角色/关系分三次独立评分 | 三者作为整体一次评分（因为相互依赖） |
| **用户预览** | 无 | 基建完成后+开篇完成后各一次用户预览确认 |
| **基建锁定** | 无 | 用户确认后锁定基建，后续不可违反 |
| **故事概览卡** | 无 | 基建评分通过后自动生成，供用户快速预览全书方向 |
| **协同度检查** | 无 | 基建评分新增 synergy_check，检查世界观/角色/关系的协同性 |
| **阶段组织** | 5个阶段线性排列 | 5个阶段组织进3大门禁阶段 |
| **开篇确认** | 评分通过即继续 | 评分通过后还需用户预览确认 |
| **锁定约束** | 无 | 已锁定基建在后续阶段中标注为🔒，强制遵守 |
| **连载生产模式** | 单章循环（写一章评一章） | 批量生成（用户指定N章/一卷/到某节点，批次完成后预览修改） |
| **批次预览** | 无 | 每批次完成后生成批次报告，用户可确认/修改指定章节/从某章重写/暂停 |

### 新增内容

| 新增内容 | 说明 |
|---------|------|
| **故事概览卡** | 基建完成后自动生成的一页摘要，包含书名/类型/世界观速览/角色表/关系线/故事主线/创新点 |
| **协同度评分** | 基建评分新增维度：世界观是否支撑角色、角色是否匹配世界观、关系是否驱动剧情、故事弧是否利用全部潜力 |
| **用户预览确认机制** | 三选一：确认锁定/局部调整/推翻重来 |
| **基建锁定规则** | 明确定义：已锁定项/允许变更项/禁止变更项/解锁流程 |
| **阶段间门禁** | 第一阶段未锁定不得进入第二阶段；第二阶段未定稿不得进入第三阶段 |
| **批量生成控制** | 用户指定生成数量（按章数/按卷/到指定节点），支持 batch_config 参数 |
| **批次报告** | 批次完成后自动生成汇总报告（章节列表/评分/摘要/亮点/问题），供用户预览 |
| **批次修改机制** | 四选一：全部确认/修改指定章节/从某章重写/暂停连载 |

### 设计动机

1. **v3.0 的线性流程缺乏"方向确认"** → 如果世界观方向错了，后面写的所有章节都是浪费
2. **基建各模块相互依赖** → 世界观决定角色空间，角色决定关系网，分开评分无法检查协同性
3. **用户应该在关键节点有决策权** → 基建方向和开篇风格是全书最重要的两个决策点
4. **需要"锁定"机制防止设定漂移** → 长篇连载中容易出现设定矛盾，锁定后强制遵守
5. **故事概览卡让用户快速判断** → 用户不需要读完所有 JSON 就能判断方向是否正确
6. **单章循环效率低且缺乏全局视角** → 用户想一次看5章或一整卷的效果，而非逐章确认；批量生成后统一预览更高效，也更容易发现跨章节的节奏问题

### 新增约束规则

**平台相关：**
- 番茄/七猫/书旗：每章2000-2300字，每2000字至少1个反转或爽点
- 起点：前3章3000字内完成"主角危机+金手指+悬念"三要素
- 知乎盐选：文笔要求高，结尾反转是标配
- 番茄日更低于4000字算法不推流

**字数相关：**
- 短篇：伏笔跨度5-15章，不超过3条暗线
- 中篇：里程碑章节体系（第10章大爽点、第30章中期转折、60%最大危机、80%终极反转）
- 长篇：每3-5卷更换场景，每2卷引入新角色，中期必须格局升级
- 超长篇：洋葱模型世界观，至少3次世界观颠覆，每50章角色状态审计

**类型相关：**
- 言情：CP必须有互补碰撞设计，禁止强制爱/PUA被美化
- 悬疑：三线索原则（每个真相至少3条线索），禁止凶手最后才出场
- 都市：打脸三次反转法，身份揭示循序渐进
- 历史：分层考据法，穿越者知识受限于当时条件

**评分相关：**
- 基建（世界观/角色/关系）评分<95分必须重写，触发红线直接重写
- 大纲评分<90分必须修改，连续3章无爽点触发红线
- 开篇评分<90分必须回炉重写，首章前500字无冲突触发红线
- 章节评分<80分必须修改，连续3章<85分触发节奏预警
- 不同类型开篇评分章数不同（番茄3章/起点5章/知乎2章）
- 不同平台评分权重不同（番茄重钩子/起点重文笔/知乎重文笔）

### 设计动机

1. **不同平台读者差异巨大** → 番茄要极快节奏，起点要深度世界观，知乎要文笔——同一个故事需要不同写法
2. **字数量级决定结构差异** → 20万字和200万字的小说在角色数量、伏笔跨度、节奏密度上完全不同
3. **v2.0 只覆盖4种类型** → 言情、悬疑、都市、历史是网文最大的几个品类，缺失严重
4. **开篇是网文生死线** → 番茄首章完读率>15%才能获得推荐，需要独立的精细化开篇系统
5. **节奏检查需要量化** → v2.0 的连贯性检查器中节奏部分太粗糙，需要独立的量化审计工具
6. **缺乏质量门禁** → 之前的流程没有"不达标就停下来"的机制，基建有问题会在后续被放大；开篇质量直接决定签约和推荐；需要分级评分+红线机制+趋势预警
